# 03 영수증 인증

> Figma node: 311:108 · 와이어프레임 이미지: `../figma/wireframes/web-wireframe-screen-1043-33__311-108.png`

## 화면 목적

행궁동 공방거리 참여 상점에서 발생한 F&B 소비 영수증을 인증하여, 오픈 스트릿 라운지(돗자리/평상/파라솔 공간) 이용 자격("티켓")을 획득하는 화면. 인증 성공 시 발급되는 토큰이 다음 화면(04 테이블 예약)에서 예약 자격 증명으로 사용된다.

인증 방법은 두 가지 중 택1:
1. 영수증 사진 촬영/업로드 → 서버 OCR로 상점·결제일 자동 검증
2. 상점 직접 선택(드롭다운) → 사용자가 상점을 수동 지정 (OCR 실패 시 폴백 경로로도 사용 가능)

## 진입 경로

- 02 메뉴 선택 화면에서 "영수증 인증하기" 등 CTA를 눌러 진입 (상단 탭 순서 01→02→03→04→05 기준)
- 상단 탭 바에서 "03 영수증 인증" 탭 직접 클릭으로도 진입 가능 (탭 바는 네비게이션 겸 현재 위치 표시. 화면별 접근 제어(예: 02를 안 거치면 03 진입 불가)는 이 와이어프레임만으로는 규정되지 않음 — 별도 확인 필요)
- 인증 완료 후에는 04 테이블 예약으로 자동 이동하거나, 재진입 시 "이미 인증됨" 상태를 보여줘야 함 (아래 "상태" 섹션 참고)

## 레이아웃 구조

모바일 화면 프레임(상태바 포함) 안에 상단→하단 세로 스택 구조:

1. **상단 네비게이션 바** (Island Yellow #FFEE00 배경)
   - 좌측: 뒤로가기 화살표 아이콘 (`←`) — 02 메뉴 선택으로 이동
   - 중앙: 화면 타이틀 "영수증 인증" (텍스트 정중앙 정렬)
2. **안내 텍스트** (본문, 회색 톤)
   - "행궁동 참여 상점의 영수증을 촬영하거나 업로드하세요. 당일 결제 영수증만 인정됩니다."
3. **영수증 업로드 드롭존** (점선 테두리, 연노랑 배경 박스, 정사각형에 가까운 큰 카드)
   - 중앙 정렬: 카메라 아이콘(원형 아웃라인 카메라 심볼)
   - 굵은 텍스트: "영수증 촬영 또는 업로드"
   - 보조 텍스트: "JPG / PNG · 최대 10MB"
   - 클릭/탭 시 카메라 촬영 또는 파일 선택(갤러리) 액션 트리거
4. **구분자**: "또는" (텍스트, 중앙 정렬, 업로드 박스와 상점 선택 드롭다운 사이)
5. **상점 직접 선택 섹션**
   - 라벨: "상점 직접 선택" (작은 회색 텍스트, 좌측 정렬)
   - 드롭다운 셀렉트: placeholder "상점을 선택하세요" + 우측 chevron-down 아이콘
6. **인증하기 버튼** (Primary CTA, 흰 배경 + 검은 테두리 or 아웃라인 버튼, 체크 아이콘 `✓` + "인증하기" 텍스트, 풀 와이드)
7. **하단 보조 텍스트**: "인증은 하루 1회만 가능합니다" (작은 회색 텍스트, 중앙 정렬)

와이어프레임 옆 "개발 스펙" 패널(디자인 요소 아님, 개발 참고용 주석)에 아래 "개발 노트" 섹션 내용이 명시되어 있음.

## 컴포넌트 상세

### 상단 네비게이션 바
- 배경색: Island Yellow `#FFEE00`
- 뒤로가기 아이콘: 클릭 시 이전 화면(02 메뉴 선택)으로 라우팅. 업로드/OCR 처리 중일 때는 비활성화 또는 확인 다이얼로그 고려

### 영수증 업로드 드롭존 (`ReceiptUploadDropzone`)
- 상태: `empty`(기본, 카메라 아이콘+안내문) / `uploading`(스피너 또는 프로그레스) / `preview`(업로드된 이미지 썸네일 표시 + 재업로드/삭제 버튼) / `error`(에러 메시지 표시, 아래 "상태" 참고)
- 파일 입력: `<input type="file" accept="image/jpeg,image/png" capture="environment">` — 모바일에서 카메라 직접 호출 지원
- 클라이언트 측 검증:
  - 파일 형식: JPG, PNG만 허용 (다른 형식 업로드 시 즉시 에러, 서버 요청 없이 차단)
  - 파일 크기: 최대 10MB (초과 시 즉시 에러, 서버 요청 없이 차단)
- 업로드된 이미지는 미리보기로 드롭존 내부에 렌더링, 우상단에 X(삭제) 버튼 노출

### 상점 선택 드롭다운 (`ShopSelect`)
- 참여 상점 목록을 서버에서 fetch (예: `GET /api/shops`) 하여 옵션으로 렌더링
- 단일 선택(single select), placeholder "상점을 선택하세요"는 미선택 상태의 disabled 옵션 취급
- 영수증 업로드와 상점 선택은 상호 배타적이지 않음: OCR이 상점을 특정하지 못할 경우 이 드롭다운으로 사용자가 직접 보정하는 용도로도 쓰일 수 있음 (OCR 결과에 상점명이 파싱되면 자동 프리필 고려)

### 인증하기 버튼 (`VerifyButton`)
- 활성화 조건: 영수증 이미지가 업로드되어 있거나, 상점이 선택되어 있어야 함 (최소 하나 이상 입력 필요). 이미지가 없고 상점만 선택된 경우의 인증 로직은 별도 API 분기 필요 — 와이어프레임 텍스트 "이미지, shop_id → {verified, token}" 기준으로 이미지가 필수인지 선택적인지 모호하므로 기획 확인 필요. 우선 구현 가이드: **이미지 업로드는 필수, shop_id는 OCR 보정용 선택값**으로 가정하고 진행하되, Tom 확인 필요
- 클릭 시: 로딩 상태로 전환 → `POST /api/verify` 호출 → 성공/실패에 따라 상태 분기
- 아이콘: 체크(`✓`) + 텍스트 "인증하기"

## 데이터

### 요청: `POST /api/verify`
- Content-Type: `multipart/form-data`
- Body 필드:
  - `image`: File (JPG/PNG, 최대 10MB)
  - `shop_id`: string | number (상점 직접 선택 시, 또는 OCR 보정용 optional)
  - 인증 주체 식별자(세션 또는 device_id)는 아래 "1회 제한" 섹션 참고 — 헤더 또는 쿠키로 전달되는 것으로 가정 (예: `Authorization` 세션 쿠키, 또는 `X-Device-Id` 헤더)
- 응답: `{ verified: boolean, token: string }`
  - `verified: true` + `token`: 인증 성공, 이 token을 다음 화면(04 테이블 예약)에서 예약 시 사용
  - `verified: false`: 인증 실패 (OCR 파싱 실패, 당일 아닌 영수증, 미참여 상점 등). 에러 사유를 구분해서 내려주는 필드(예: `reason: "not_today" | "ocr_failed" | "shop_not_participating" | "already_verified"`)가 필요 — 현재 와이어프레임 스펙에는 명시되어 있지 않으므로 백엔드와 협의 필요

### 참여 상점 목록 조회 (드롭다운용, 엔드포인트 미정)
- 가정: `GET /api/shops` → `[{ id, name }, ...]`
- 02 메뉴 선택 화면에서 이미 상점 목록을 다루고 있을 가능성이 높으므로, 해당 화면 스펙과 데이터 소스를 통일할 것 (중복 API 만들지 말고 재사용 검토)

## 상태

1. **초기 상태 (empty)**: 업로드 박스 비어있음, 상점 미선택, 인증하기 버튼 비활성 또는 클릭 시 유효성 에러
2. **이미지 업로드 완료 (preview)**: 드롭존에 썸네일 표시, 인증하기 버튼 활성화
3. **인증 처리 중 (loading)**: 인증하기 버튼 로딩 스피너로 전환, 중복 클릭 방지(버튼 disable)
4. **인증 성공**: `verified: true` → token을 로컬 저장(localStorage/sessionStorage 또는 전역 상태)하고 04 테이블 예약 화면으로 자동 이동
5. **인증 실패 — 재시도 허용 (error, retryable)**: 에러 메시지를 드롭존 하단 또는 업로드 박스 내부에 표시 (예: "영수증을 다시 확인해주세요" / "오늘 결제한 영수증만 인정됩니다" / "참여 상점 영수증이 아닙니다"). 이미지 재업로드 또는 상점 재선택 후 재시도 가능 — 재시도 횟수 제한은 명시되어 있지 않음(무제한으로 가정)
6. **이미 인증됨 (already verified, 1일 1회 제한 초과)**: 하루 1회 제한에 걸린 사용자가 재진입 시 — 업로드 폼 대신 "오늘 이미 인증하셨습니다" 안내와 함께 보유 중인 token 정보 또는 04 화면으로 바로 이동하는 CTA를 보여주는 것이 자연스러움. 와이어프레임에는 이 상태의 UI가 별도로 그려져 있지 않으므로 구현 시 추가 설계 필요 (Tom 확인 필요)
7. **당일 유효성 실패**: 영수증의 결제일(OCR로 파싱된 날짜)이 오늘(서버 기준 today)과 다르면 실패 처리. 서버 사이드 validation이며 클라이언트에서 미리 판단 불가

## 이동

- 좌상단 뒤로가기 (`←`) → 02 메뉴 선택 화면
- 상단 탭 바 "01 메인" / "02 메뉴 선택" / "04 테이블 예약" / "05 방명록" 클릭 → 해당 화면으로 직접 이동 (탭 네비게이션. 04, 05 탭이 인증 전에도 클릭 가능한지, 인증 완료 전 접근을 막아야 하는지는 별도 라우트 가드 정책 필요)
- 인증 성공(`verified: true`) → 04 테이블 예약 화면으로 자동 이동, token을 쿼리 파라미터/전역 상태/스토리지로 전달

## 개발 노트

와이어프레임 옆 "개발 스펙" 패널에 명시된 기술 사양(원문 반영):

- **이미지 업로드**: `multipart/form-data` 방식으로 서버 전송
- **OCR 처리**: 서버사이드에서 영수증 이미지 파싱 (클라이언트에서 OCR 하지 않음). OCR 엔진/라이브러리 선정은 백엔드 영역 — 별도 결정 필요
- **당일 유효**: "결제일 = 오늘" validation. OCR로 추출한 영수증 결제 일자와 서버의 현재 날짜(서버 타임존 기준)를 비교해 당일이 아니면 인증 실패 처리
- **실패 처리**: 에러 메시지를 노출하고 재시도를 허용한다 (인증 시도 자체의 재시도 제한은 없음 — 단, 아래 "1회 제한"과는 별개 개념. 즉 하루 인증 성공은 1회지만, 인증 실패 후 재시도는 여러 번 가능)
- **1회 제한**: 인증은 "session 또는 device_id 기반"으로 하루 1회만 허용. 로그인 시스템이 없는 것으로 보이므로(영수증 인증제가 핵심 진입점), 익명 사용자를 device_id(브라우저 fingerprint, localStorage에 생성한 UUID 등) 또는 서버 세션 쿠키로 식별하여 서버 측에서 하루 1회 제한을 강제해야 함. 클라이언트 로컬 스토리지만으로 제한하면 우회가 쉬우므로 **서버 사이드에서 device_id/session 기준으로 반드시 재검증**할 것
- **API**: `POST /api/verify`
  - 요청: `image`, `shop_id`
  - 응답: `{ verified: boolean, token: string }`

추가로 구현 에이전트가 참고할 미확정 사항 (Tom 확인 필요):
- `shop_id`가 이미지 업로드와 함께 필수 파라미터인지, OCR 실패 시에만 쓰이는 optional 파라미터인지
- 인증 실패 시 `reason` 코드 체계 (사유별 메시지 분기를 위해 필요)
- "이미 인증됨(하루 1회 초과)" 상태의 화면 UI
- device_id 생성/저장 전략 (UUID 생성 시점, localStorage 키 이름, 서버와의 매칭 방식)
- 04 테이블 예약 화면에서 token을 어떤 형태로 전달받는지 (쿼리 파라미터 vs 전역 상태 vs 재조회)
