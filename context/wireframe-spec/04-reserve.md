# 04 테이블 예약

> Figma node: 311:107 · 와이어프레임 이미지: `../figma/wireframes/web-wireframe-screen-1043-37__311-107.png`

## 화면 목적

영수증 인증(03 영수증 인증)을 마친 사용자가 오픈 스트릿 라운지의 빈 좌석(테이블)을 실시간 현황판에서 골라 예약을 확정하는 화면. 예약이 확정되면 좌석 사용 권한을 증명하는 QR 토큰이 발급되고, 이후 방명록(05) 등 후속 플로우로 이어진다.

## 진입 경로

- **03 영수증 인증** 화면에서 영수증 인증 성공(`verify_token` 발급) 직후 자동 전환되는 것이 유일한 진입 경로. 인증 없이 이 화면에 직접 진입하는 것은 허용하지 않는다(가드 필요, 아래 "상태" 참고).
- 진입 시 이전 화면에서 다음 파라미터를 전달받아야 함:
  - `verify_token` (string) — 영수증 인증 완료 토큰. 예약 확정 API 호출 시 그대로 사용.
  - `cafe_name` (string) — 인증된 매장명. 상단 배너 "카페 A 영수증 인증 완료"의 "카페 A" 부분에 바인딩.

## 레이아웃 구조

상단 5단 탭 네비게이션(01 메인 / 02 메뉴 선택 / 03 영수증 인증 / **04 테이블 예약**(활성, 두꺼운 테두리로 강조) / 05 방명록) 아래, 모바일 프레임(375~390px 폭 기준) 세로 스택 구조:

1. 상태바 (9:41, 더미 — 실 서비스에는 렌더링 불필요)
2. 헤더 바 (Island Yellow 배경, 좌측 뒤로가기 화살표 + 중앙/좌측 정렬 타이틀 "자리 선택")
3. 인증 완료 배너 (풀블리드 옐로, 체크 아이콘 + "{cafe_name} 영수증 인증 완료")
4. 좌석 상태 범례 (체크박스 아이콘 3종 + 라벨: 선택 / 예약 가능 / 사용 중)
5. 좌석 그리드 — 2행 × 3열 카드 (A1, A2, A3 / B1, B2, B3)
6. 선택된 자리 요약 박스 (회색 배경 카드)
7. CTA 버튼 "예약 확정하기" (캘린더 아이콘 + 텍스트, 풀 위드 화이트 버튼 + 보더)

## 컴포넌트 상세

### 1. 헤더 바
- 배경: Island Yellow (#FFEE00)
- 좌측: 뒤로가기 화살표 아이콘 (`←`) — 탭 시 03 영수증 인증 또는 이전 화면으로 복귀. 단, 인증 완료 후 뒤로가기 시 `verify_token`을 폐기할지 유지할지는 기획 확인 필요(기본값: 유지, 재진입 허용).
- 중앙/좌측: 타이틀 텍스트 "자리 선택" (고정 문자열)

### 2. 인증 완료 배너
- 배경: Island Yellow, 풀블리드(카드 좌우 인셋 없이 헤더 바로 아래 가로 전체)
- 아이콘: 체크(✓) — Flow Black
- 텍스트: `"{cafe_name} 영수증 인증 완료"` (예: "카페 A 영수증 인증 완료")
- 이 배너는 정적 표시 요소이며 인터랙션 없음. `verify_token`이 없거나 만료된 경우 이 화면 진입 자체를 막아야 하므로(아래 "상태" 참고) 배너가 없는 상태는 존재하지 않음.

### 3. 좌석 상태 범례
가로 배치, 좌측부터:
- 체크박스(옐로 채움, 체크 표시) + "선택" — 좌석 카드가 `selected` 상태일 때의 스타일 예시
- 체크박스(옐로 테두리, 빈 채움) + "예약 가능" — `available` 상태 스타일 예시
- 체크박스(회색 채움) + "사용 중" — `taken` 상태 스타일 예시

이 범례는 순수 UI 가이드이며 클릭 불가. 실제 좌석 카드 스타일과 1:1 매핑되어야 함(색상 토큰은 아래 데이터/상태 섹션 참고).

### 4. 좌석 그리드
- 레이아웃: 2행 3열, 카드 간 동일 gap
- 카드 구성 요소:
  - 좌석 라벨 (예: "A3") — bold, 카드 내 상단
  - 서브 텍스트 — 두 종류 중 하나만 표시:
    - 정원 표기: "2인석" / "4인석" (available 또는 selected 상태일 때)
    - 상태 표기: "사용 중" (taken 상태일 때, 정원 정보 대신 상태 텍스트로 대체 표시)
- 좌석별 예시 데이터 (와이어프레임 기준):
  | seat_id | capacity | status | 표시 서브텍스트 |
  |---|---|---|---|
  | A1 | 2 | taken | 사용 중 |
  | A2 | 2 | available | 2인석 |
  | A3 | 4 | selected | 4인석 |
  | B1 | 2 | available | 2인석 |
  | B2 | 2 | taken | 사용 중 |
  | B3 | 4 | available | 4인석 |
- 카드 스타일 (상태별):
  - `available`: 배경 연한 크림/화이트, 옐로 테두리(thin), 텍스트 Flow Black, 탭 가능(pointer)
  - `taken`: 배경 회색(#EEEEEE 톤), 테두리 없음 또는 옅은 회색, 텍스트 회색(비활성), **탭 불가** (disabled, onClick 무시 + 시각적 disabled 커서)
  - `selected`: 배경 Island Yellow 풀 채움, 굵은 검정 테두리, 텍스트 Flow Black bold — 그리드 내 **동시에 최대 1개**만 이 상태 가능

### 5. 선택된 자리 요약 박스
- 배경: 연회색 카드, 그리드 하단에 위치
- 라벨: "선택된 자리" (고정, 작은 글씨)
- 값: `"{seat_id} · {capacity}인석 · {position_label}"` 형식 (예: "A3 · 4인석 · 창가 자리")
  - `position_label`은 좌석 메타데이터에 포함된 위치 설명 문자열 (예: "창가 자리", "중앙 자리" 등) — API 응답(`seats[]`)에 필드로 포함되어야 함. 와이어프레임에는 A3만 "창가 자리"로 예시 표기되어 있어 좌석마다 다른 라벨이 있을 수 있음을 시사.
- **좌석 미선택 상태**: 와이어프레임에는 이 상태가 그려져 있지 않으나, 화면 진입 초기값은 미선택이어야 하므로 다음을 구현 규칙으로 정한다:
  - 미선택 시 이 박스는 "선택된 자리가 없습니다" 등 플레이스홀더 텍스트를 표시하거나, 박스 자체를 숨김 처리
  - CTA 버튼은 미선택 상태에서 **비활성화**(disabled)

### 6. CTA 버튼 "예약 확정하기"
- 아이콘: 캘린더
- 풀 위드, 화이트 배경 + 검정 보더 (버튼 스타일은 디자인 시스템 공용 primary/secondary 버튼 규칙을 따르되, 색상 결정은 디자인 팀 확인 필요 — 와이어프레임은 흑백 목업이라 실제 컬러 미지정)
- 활성 조건: 좌석이 `selected` 상태로 정확히 1개 선택되어 있을 때만 활성화(탭 가능)
- 탭 시 동작: `POST /api/reserve` 호출 → 성공 시 다음 화면(QR 발급/확인 화면 또는 05 방명록)으로 전환. 실패 시 에러 처리(아래 "상태" 참고)

## 데이터

### 좌석 목록 조회
```
GET /api/seats
```
- Response: `seats[]` 배열. 각 항목 스키마(추정, 백엔드 확정 필요):
```json
{
  "seat_id": "A3",
  "capacity": 4,
  "position_label": "창가 자리",
  "status": "available" // "available" | "taken"
}
```
  - 주의: `selected`는 서버 상태값이 아니라 **클라이언트 로컬 UI 상태**다. 서버는 `available` / `taken`만 내려주고, 사용자가 좌석을 탭하면 프론트에서 해당 좌석에 한해 `selected`로 표시한다. 다른 사용자가 동시에 그 좌석을 선점하면(taken으로 바뀌면) 실시간 업데이트로 즉시 반영하고 로컬 선택은 무효화해야 한다(아래 "상태" 참고).

### 예약 확정
```
POST /api/reserve
Request body: { "seat_id": string, "verify_token": string }
Response: { "reservation_id": string, "qr_token": string }
```
- `verify_token`은 03 영수증 인증에서 전달받은 값을 그대로 사용
- 성공 시 `reservation_id`를 로컬(예: sessionStorage 또는 앱 상태)에 저장 — 이후 화면(방명록 등)에서 예약 확인용으로 참조 가능
- `qr_token`은 좌석 사용 권한을 증명하는 QR 코드 생성에 사용 (QR 표시 화면은 이 와이어프레임 범위 밖 — 05 방명록 진입 전 별도 확인 화면 존재 여부는 기획 확인 필요)

## 상태

1. **초기 로딩**: `GET /api/seats` 응답 대기 중 — 그리드 스켈레톤 또는 로딩 인디케이터 표시(와이어프레임에 명시 없음, 구현 시 skeleton 권장)
2. **좌석 미선택 (기본)**: 진입 시 아무 좌석도 `selected` 아님. CTA 버튼 disabled.
3. **좌석 선택됨**: 사용자가 `available` 좌석 카드를 탭 → 해당 좌석 `selected`로 전환, 이전에 선택돼 있던 다른 좌석은 `available`로 롤백(라디오 버튼처럼 상호 배타적, 스펙 명시: "좌석 선택: 하나만 선택 toggle"). 같은 좌석을 다시 탭하면 선택 해제(toggle off) — CTA 버튼 다시 disabled.
4. **`taken` 좌석 탭**: 무시(비활성 카드는 클릭 이벤트 자체를 받지 않도록 구현)
5. **실시간 동시성 처리**: 다른 사용자가 좌석을 선점하는 경우를 대비해 WebSocket 또는 SSE로 `seats[]` 상태를 구독. 서버에서 특정 seat_id의 status가 `taken`으로 바뀌었다는 이벤트 수신 시:
   - 해당 좌석 카드를 즉시 `taken` 스타일로 갱신
   - 만약 그 좌석이 현재 사용자가 `selected`한 좌석이었다면 선택을 강제 해제하고 안내 메시지 표시(예: "다른 사용자가 먼저 예약했습니다") — 메시지 UX는 구현 시 재량, 단 반드시 사용자에게 알려야 함(무음 처리 금지)
6. **예약 확정 요청 중**: CTA 버튼 탭 후 `POST /api/reserve` 응답 대기 — 버튼 로딩 스피너 표시, 중복 탭 방지(버튼 비활성화)
7. **예약 확정 성공**: `reservation_id`, `qr_token` 수신 → 다음 화면으로 이동(아래 "이동" 참고)
8. **예약 확정 실패** (예: 이미 다른 사용자가 선점, verify_token 만료 등): 에러 토스트/모달 표시 후 좌석 목록 재조회(`GET /api/seats`)하여 최신 상태로 그리드 갱신, 선택 상태 초기화
9. **인증 가드**: `verify_token`이 없거나 만료된 상태로 이 화면에 직접 진입 시도 시 → 03 영수증 인증 화면으로 강제 리다이렉트 (라우트 가드 필요)

## 이동

- **뒤로가기(←)**: 03 영수증 인증(또는 진입 직전 화면)으로 복귀
- **예약 확정 성공**: 다음 화면으로 전환. 와이어프레임 세트에 예약 확정 후 화면이 별도로 존재하지 않으므로, QR 토큰을 보여주는 확인 화면이 있는지 또는 바로 **05 방명록**으로 넘어가는지는 기획 확인 필요(디자인팀 확인 항목). 백엔드 응답에 이미 `qr_token`이 포함되므로 프론트는 최소한 QR을 노출할 화면/모달을 어딘가에는 배치해야 함.

## 개발 노트

- 좌석 상태 3종(`available` / `taken` / `selected`)에서 `selected`는 서버 상태가 아닌 클라이언트 로컬 상태임을 반드시 구분해서 구현할 것 — 서버 스키마에 `selected`를 넣지 말 것.
- 실시간 업데이트 방식은 WebSocket 또는 SSE 둘 다 명시되어 있어 택일 가능 — 다만 팀 백엔드 스택(기존 API가 REST 기반)을 고려하면 SSE가 구현 부담이 적음. 최종 선택은 백엔드 담당자 확인 필요(설계 결정 사항, 임의 확정 금지).
- 좌석 선택은 라디오 버튼과 동일한 단일 선택 토글 — 별도 "여러 명 좌석 조합 예약" 같은 멀티 셀렉트 요구사항은 이 와이어프레임에 없음. 향후 요구사항 추가 시 구조 변경 필요.
- `position_label`(창가 자리 등) 필드는 좌석 마스터 데이터에 미리 등록돼 있어야 함 — 시드 데이터 설계 시 6개 좌석(A1~B3) 각각의 capacity, position_label을 확정해야 함(디자인/운영팀 확인 필요, 목공방 협업 가구 배치도 참고).
- CTA 버튼 색상(화이트+블랙보더 vs Island Yellow 채움)은 와이어프레임이 흑백 목업이라 실제 컬러 확정 안 됨 — 디자인 시스템(Island Yellow / Pause White / Flow Black) 적용 시 디자인팀 확인 후 반영.
- 예약 확정 후 화면 전환 목적지(QR 노출 화면 존재 여부)는 이 스펙만으로 확정 불가 — 05 방명록 와이어프레임과 교차 확인 필요.
