# 05 방명록

> Figma node: 311:106 · 와이어프레임 이미지: `../figma/wireframes/web-wireframe-screen-1043-40__311-106.png`

## 화면 목적

방문자가 익명으로 공간 이용 후기를 남기고, 다른 방문자들이 남긴 후기를 시간순으로 볼 수 있는 화면. "영수증 인증제"(주변 F&B 소비 = 공간 이용 티켓)와 연계해서, 방문자가 어느 맛집/상점을 이용했는지 태그로 남길 수 있게 하여 상권 매출-체류 연결 스토리를 아카이빙하는 역할을 겸한다. 로그인/회원 시스템 없이 익명 게시판 형태로 운영한다.

## 진입 경로

- 하단(또는 상단) 5단계 플로우 탭 바에서 "05 방명록" 탭 선택 시 진입 (01 메인 → 02 메뉴 선택 → 03 영수증 인증 → 04 테이블 예약 → 05 방명록 순서의 마지막 단계)
- 04 테이블 예약 완료 후 자연스럽게 다음 단계로 넘어오는 것으로 추정 (별도 확인 필요 — 예약 완료와 무관하게 언제든 방명록 탭으로 직접 진입 가능한 구조로 보임)
- 화면 상단 back 버튼(←)으로 이전 화면(04 테이블 예약 또는 진입 전 화면)으로 복귀

## 컴포넌트 상세

### 1. 헤더 바 (`GuestbookHeader`)
- 배경: Island Yellow `#FFEE00`

### 2. 후기 작성 폼 (`GuestbookComposer`)
- **텍스트 입력창** (`<textarea>` 또는 멀티라인 TextField)
  - Placeholder: "오늘 카페 A에서 커피 사고 여기서 책 읽었어요. 너무 좋은 공간이에요..."
  - 필수 입력, 최소 1자 이상 (공백만 있는 값은 제출 불가하도록 trim 검증)
  - 최대 길이 제한 필요 (권장: 200자, 개발 시 확정 필요 — Figma에 명시 없음)
  - 둥근 사각형 테두리, 배경 흰색
- **맛집 추가 버튼** (`ShopTagButton`)
  - 아이콘: 위치 핀(pin) 아이콘 + 텍스트 "맛집 추가"
  - 탭 시 상점 목록 선택 UI(바텀시트 또는 모달) 오픈
  - 개발 스펙 명시: **"상점 목록 연동 선택"** — 자유 텍스트 입력이 아니라 사전 정의된 상점(F&B 파트너) 목록에서 선택하는 방식. 즉 별도 API(`GET /api/shops` 등, 스펙에 명시 없으나 존재 가정 필요 — 확인 필요)로 상점 리스트를 불러와 사용자가 선택
  - 복수 선택 가능 (POST 요청 바디가 `shop_tags[]` 배열이므로 다중 태그 지원)
  - 선택된 태그는 버튼 아래 또는 버튼 내부에 칩(chip) 형태로 표시되어야 함 (와이어프레임에는 선택 후 상태 미표현 — 확인 필요)
- **별점 추가 버튼** (`RatingButton`)
  - 아이콘: 별(★) + 텍스트 "별점 추가"
  - 탭 시 별점 선택 UI 노출 (1~5점 등, 척도는 와이어프레임에 명시 없음 — 확인 필요)
  - 개발 스펙 패널에는 별점 관련 API 필드가 명시되어 있지 않음 (`content`, `shop_tags[]`만 언급). 별점 데이터를 POST 바디에 포함할지, 혹은 이번 스코프에서 제외할지 **Tom 확인 필요**
- **게시하기 버튼** (`SubmitButton`)
  - 아이콘: 종이비행기(send) + 텍스트 "게시하기"
  - 전체 폭, 검은색 또는 강조색 배경 (와이어프레임은 outline 스타일로 보임 — 실제 색상은 디자인 팀 확정 필요)
  - 탭 시 `POST /api/guestbook` 호출
  - 제출 중 로딩 상태(버튼 비활성화 + 스피너) 필요
  - 성공 시: 입력창 초기화, 목록 최상단에 새 항목 optimistic하게 추가 또는 목록 재조회
  - 실패 시: 에러 토스트/인라인 메시지 노출, 입력 내용 유지

### 3. 최근 방명록 섹션 (`GuestbookList`)
- 섹션 라벨: "최근 방명록" (좌측 정렬, 작은 텍스트)
- **방명록 항목 카드** (`GuestbookEntryCard`), 리스트 반복 렌더링
  - 상단 좌측: 작성자 표시 "익명 방문자" (고정 텍스트, 실제 닉네임/식별자 없음 — 익명 처리)
  - 상단 우측: 상대 시간 텍스트 (예: "10분 전", "42분 전") — 서버의 `created_at`을 클라이언트에서 상대 시간으로 변환 (예: dayjs relativeTime 또는 date-fns formatDistanceToNow)
  - 본문: 후기 내용 텍스트 (여러 줄 가능, 줄바꿈 유지)
  - 배경: 연한 회색 라운드 카드
  - 맛집 태그가 있는 항목의 경우 태그 chip을 카드 내 표시할 것으로 예상되나 와이어프레임 목업 데이터에는 태그 UI가 노출되어 있지 않음 (텍스트 본문에 "서점 B", "음식점 C" 등이 자연어로만 녹아 있음) — **태그 표시 UI는 별도 확정 필요**
- 리스트는 최신순 정렬(`created_at DESC`)로 렌더링
- **Infinite scroll**: 리스트 하단 도달 시 다음 페이지 자동 로드, 페이지당 10개씩
  - 로딩 중에는 리스트 하단에 스피너 표시
  - 더 이상 데이터가 없으면 "마지막 항목입니다" 등 종료 표시 (선택 사항, 확인 필요)

## 데이터

### 방명록 항목 (`GuestbookEntry`)
| 필드 | 타입 | 설명 |
|---|---|---|
| `entry_id` | string | 항목 고유 ID, `POST` 응답으로 반환됨 |
| `content` | string | 후기 본문 텍스트 |
| `shop_tags` | string[] | 태그된 상점 ID 배열 (0개 이상) |
| `created_at` | string (ISO 8601) | 작성 시각, 정렬 기준 |

작성자 관련 필드는 응답에 포함되지 않거나 포함되더라도 클라이언트에는 노출하지 않음 — UI에는 항상 "익명 방문자"로 고정 표시. 서버 측에서 스팸/어뷰징 방지를 위해 내부적으로 세션/디바이스 식별자를 저장할 수는 있으나 응답 스키마에는 노출 금지 (익명성 보장).

## 상태

- **입력 대기 상태**: 텍스트 입력창 비어있음, 게시하기 버튼 비활성화 또는 활성화(빈 값 제출 시 검증 에러) — 구현 시 버튼을 비활성화하는 방식 권장
- **맛집 태그 선택 상태**: 0개 선택(기본) / 1개 이상 선택(칩 노출)
- **제출 중 상태**: 게시하기 버튼 로딩 스피너, 중복 제출 방지(디바운스/버튼 비활성화)
- **제출 성공 상태**: 입력 폼 초기화(텍스트, 태그, 별점 모두 리셋), 목록 갱신
- **제출 실패 상태**: 에러 메시지 노출(예: "게시에 실패했습니다. 다시 시도해주세요."), 입력값 보존
- **목록 로딩 상태 (최초)**: 스켈레톤 UI 또는 스피너
- **목록 로딩 상태 (추가 페이지)**: 리스트 하단 스피너
- **목록 빈 상태**: 항목이 0개일 때 "아직 등록된 방명록이 없어요" 등 빈 상태 문구 (와이어프레임에 명시 없음, 확인 필요)
- **네트워크 에러 상태**: 목록 로드 실패 시 재시도 버튼 노출

## 이동

- 헤더 back 버튼 → 이전 화면(04 테이블 예약 혹은 진입 이전 화면)으로 복귀
- 상단 탭 바의 다른 단계(01~04) 탭 선택 시 해당 화면으로 즉시 이동 (전역 네비게이션으로 보임, 실제 구현 시 스텝 인디케이터가 아닌 자유 이동 탭인지 Tom 확인 필요 — 통상적인 플로우라면 순차 진행이 맞을 수 있음)
- 맛집 추가 버튼 → 상점 선택 바텀시트/모달 오픈 (별도 화면 전환 없이 오버레이로 처리 권장)
- 별점 추가 버튼 → 별점 선택 UI 오픈 (오버레이 처리 권장)
- 게시하기 성공 → 같은 화면 유지, 목록 갱신 (별도 화면 이동 없음)

## 개발 노트

- **API 엔드포인트**
  - `POST /api/guestbook`
    - Request body: `{ content: string, shop_tags: string[] }`
    - Response: `{ entry_id: string }`
    - 별점(`rating`) 필드가 스펙 패널에 명시되지 않음 — "별점 추가" UI가 있으나 서버 저장 여부 불명확. 구현 전 Tom에게 별점 데이터 처리 방식 확인 필요 (저장 안 함 / body에 추가 필드로 포함 등)
  - `GET /api/guestbook`
    - Query params: `cursor` (다음 페이지 조회용, infinite scroll 커서 기반 페이지네이션), `limit` (기본 10)
    - Response: `{ entries: GuestbookEntry[], cursor: string | null }` — `cursor`가 `null`이면 마지막 페이지
- **정렬**: 서버에서 `created_at DESC`로 정렬해서 내려줌. 클라이언트는 별도 정렬 로직 불필요, 받은 순서 그대로 렌더링
- **페이지네이션**: offset 기반이 아닌 **cursor 기반 infinite scroll**, 페이지당 10개. Intersection Observer 등으로 리스트 하단 감지 시 다음 `cursor`로 `GET` 재호출
- **익명 처리**: 로그인/회원 시스템 없음. 서버는 작성자 식별 정보(닉네임 등)를 응답에 포함하지 않으며, 클라이언트는 모든 항목에 "익명 방문자"를 하드코딩 표시. 스팸 방지가 필요하다면 서버 단에서 IP/디바이스 기반 rate limiting 고려 (명시적 스펙 없음, 향후 논의 필요)
- **맛집 태그 데이터 소스**: "상점 목록 연동 선택"이라는 문구로 보아, 이 프로젝트의 F&B 파트너(막걸리계보 등) 및 참여 상점 마스터 데이터가 별도로 존재해야 함. 해당 상점 목록을 가져오는 API(예: `GET /api/shops`)가 이 화면 구현의 선행 조건이며 현재 스펙 문서에는 없음 — 존재 여부 및 데이터 구조 확인 필요
- **상대 시간 포맷**: "10분 전", "42분 전" 형식. `created_at`(ISO 8601)을 기준으로 클라이언트에서 실시간 계산, 라이브러리 사용 권장(dayjs relativeTime 플러그인 등)
- **텍스트 최대 길이**: 와이어프레임/스펙 패널에 명시 없음. 서버 DB 컬럼 타입(TEXT vs VARCHAR)과 함께 확정 필요
- **XSS 방지**: `content`는 사용자 입력 자유 텍스트이므로 렌더링 시 반드시 이스케이프 처리 (React 기본 이스케이프로 충분하나, dangerouslySetInnerHTML 등 사용 금지)
- **낙관적 업데이트(optimistic update)**: 게시 성공 시 서버 재조회 없이 응답받은 `entry_id`로 임시 항목을 목록 최상단에 삽입하는 방식 고려 가능. 단 `created_at`이 응답에 없으므로 클라이언트 시각을 임시로 사용하거나, 재조회 방식으로 단순화하는 것을 권장 (확인 필요)
