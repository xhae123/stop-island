# 07 — 시스템 시나리오 (SYS)

> **한 줄 요약:** 화면 뒤에서 도는 규칙들 — 예약 만료, KST 일일 경계, device_id 식별, 동시성 방어, 어뷰징 수용선, 데이터 수명, 인프라 복구, 시간대 검증.

**커버 범위**
- 예약 만료 (SYS-1xx) — D-11
- 일일 경계 (SYS-2xx) — D-01, D-08
- device_id 식별 (SYS-3xx) — D-07
- 동시성 (SYS-4xx) — D-12, design-receipt-ocr의 check_and_mark
- 어뷰징과 수용 리스크 (SYS-5xx)
- 데이터 수명 (SYS-6xx) — D-23
- 인프라 복구 (SYS-7xx) — D-25 연결
- 시간대 (SYS-8xx) — D-01

## 예약 만료 (D-11)

> 📌 결정: **만료 판정 부등식은 `expires_at <= now`.** 경계 시각 정확히 그 순간은 "만료됨"이다. lazy 판정과 스윕이 같은 부등식을 쓰므로 두 경로의 판정이 절대 어긋나지 않는다.

> 📌 결정: **스윕 구현은 FastAPI 프로세스 내 asyncio 태스크(60초 간격), 쿼리는 단건 UPDATE.** `UPDATE reservations SET status='expired' WHERE status='active' AND expires_at <= now`. 별도 스케줄러(cron·celery)는 과설계 — 앱 기동 시 태스크 시작, 프로세스와 생사를 같이한다(재시작 영향은 SYS-703).

#### SYS-101 lazy 판정 — 조회 시점 만료 처리 (정상)
- Given: A1의 active 예약이 expires_at을 5초 전에 넘겼고, 다음 스윕까지 55초 남았다
- When: 어떤 사용자가 `GET /api/status` 또는 `GET /api/seats`를 호출한다
- Then: 조회 핸들러가 응답 계산 전에 만료 UPDATE를 먼저 실행 — 해당 예약이 `expired`로 바뀐다
- And: 응답의 빈자리 수·좌석 상태에 A1이 빈자리로 정확히 반영된다 (스윕을 기다리지 않음)

#### SYS-102 백그라운드 스윕 — 조회 없이도 만료 (정상)
- Given: 아무도 API를 호출하지 않는 한산한 시간, expires_at이 지난 active 예약 2건
- When: 60초 주기 스윕 태스크가 돈다
- Then: 2건 모두 `expired`로 일괄 UPDATE — 다음 조회는 lazy 처리 없이 이미 정확하다
- And: 원 예약자가 나중에 `GET /api/reservations/:id`를 하면 expired를 받고, 예약 완료 화면은 "예약이 만료됐어요" 상태로 전환된다. verify token은 당일이면 재예약에 재사용 가능 (D-08)

#### SYS-103 expires_at 정확히 경계 시각 (경계)
- Given: 예약의 expires_at이 14:00:00.000이다
- When: 정확히 14:00:00.000에 판정(lazy 또는 스윕)이 실행된다
- Then: `expires_at <= now`이므로 만료로 처리된다 — 두 경로 모두 동일 판정
- And: 13:59:59.999의 조회에서는 active로 유지된다

#### SYS-104 스윕과 사용자 조회 동시 발생 (동시성)
- Given: 만료 대상 예약 1건. 스윕 UPDATE와 사용자 조회의 lazy UPDATE가 동시에 실행된다
- When: 둘 다 같은 UPDATE 문(`WHERE status='active' AND expires_at <= now`)을 실행한다
- Then: SQLite의 단일 writer 직렬화로 한쪽이 먼저 반영되고, 다른 쪽은 매칭 0행으로 no-op — 어느 순서든 최종 상태는 `expired` 1회 전이로 동일 (멱등)
- And: 사용자 응답은 두 경우 모두 만료 반영 후의 정확한 빈자리 수를 담는다

#### SYS-105 만료 처리 도중 프로세스 재시작 (복구)
- Given: 스윕이 돌기 직전 서버가 재시작됐다
- When: 앱이 다시 기동한다
- Then: expires_at은 DB에 영속돼 있으므로 유실 없음 — 기동 후 첫 스윕 또는 첫 lazy 조회가 밀린 만료를 전부 처리한다
- And: 만료가 "최대 재시작 공백 + 60초"만큼 늦게 반영될 수 있으나, lazy 판정이 조회 경로를 항상 보정하므로 사용자가 보는 값은 틀리지 않는다

- 오류: 해당 없음 — 만료는 서버 내부 상태 전이라 사용자 입력 오류 케이스가 없다. UPDATE 실패는 SQLite 장애 급이며 SYS-7xx 범주.

## 일일 경계 (D-01, D-08)

> 📌 결정: **자정에 걸친 active 예약은 2시간 그대로 유지한다 (자정 컷 안 함).** 근거: 예약은 "좌석 점유"이고 토큰은 "예약할 권리"다. 무효화해야 하는 건 권리(토큰)지 이미 앉아 있는 사람이 아니다. 23:30 예약자를 자정에 쫓아내면 현장 갈등만 생긴다. 단, 팝업 운영시간 종료(현장 철수)는 시스템이 아니라 운영진이 정리하는 영역이다.

> 📌 결정: **토큰 무효화는 배치 없이 lazy 판정.** 토큰 사용/조회 시점에 `verified_at의 KST 날짜 == 오늘`을 검사한다. 자정에 도는 무효화 잡은 없다 — 검사가 모든 사용 경로에 있으면 잡은 불필요한 중복이다.

#### SYS-201 자정 경과 후 어제 토큰으로 예약 시도 (정상)
- Given: 어제 23:00에 approved된 토큰을 가진 사용자
- When: 오늘 00:10에 `POST /api/reserve`를 호출한다
- Then: 서버가 토큰의 verified_at KST 날짜가 오늘이 아님을 확인 → 401 + `{ error: { code: "TOKEN_EXPIRED" } }`
- And: 프론트는 라우트 가드(D-14)에 따라 `/verify`로 보낸다 — "오늘의 영수증으로 다시 인증해주세요"
- And: DB의 verification 행 자체는 변경 없음 (상태를 안 바꾸고 판정만 날짜로)

#### SYS-202 자정 직전 발급된 토큰 (경계)
- Given: 23:59:50에 approved된 토큰
- When: 23:59:59에 예약하면
- Then: 당일 토큰이므로 정상 예약된다
- When: 00:00:01에 예약하면
- Then: `TOKEN_EXPIRED` — 발급 10초 만에 만료가 맞는 동작이다. "영수증 하나 = 하루 이용권"(D-08)의 자정 경계를 예외 없이 적용한다. 자정 직전 인증하는 사용자는 팝업 운영시간(주간~저녁)상 사실상 없어 유예 로직은 만들지 않는다

#### SYS-203 자정에 걸친 active 예약 (경계)
- Given: 23:30에 시작된 active 예약 (expires_at 01:30)
- When: KST 자정이 지난다
- Then: 예약은 아무 영향 없이 active 유지 — 01:30까지 좌석을 점유한다 (위 📌 결정)
- And: 좌석 현황·빈자리 수에도 계속 반영된다
- And: 예약자의 토큰은 자정부로 만료됐으므로, 이 예약이 끝난 뒤 재예약하려면 오늘 영수증으로 재인증해야 한다

#### SYS-204 today_visitors 자정 리셋 (정상)
- Given: 어제 approved 인증 30건
- When: 자정 이후 `GET /api/status`를 호출한다
- Then: `today_visitors = 0`부터 다시 센다 — 저장된 카운터가 아니라 `verified_at의 KST 날짜 == 오늘 AND status = approved` 집계 쿼리(D-20)이므로 "리셋"이라는 상태 전이 자체가 없다. 배치도 없고 리셋 실패 케이스도 없다

#### SYS-205 자정 정각을 사이에 둔 인증 제출 (동시성)
- Given: 사용자가 23:59:59에 사진을 제출했고 OCR 처리가 자정을 넘겨 끝났다
- When: 서버가 승인 판정을 내린다
- Then: DateVerifier(D-03)의 "당일" 기준은 판정 시점의 KST 오늘이므로, 어제 날짜 영수증은 reject된다 — 판정 시각 단일 기준이라 경계 모호함이 없다
- And: 사용자는 "오늘 영수증이 아니에요" 사유를 받는다. 억울한 초 단위 경계 케이스는 manual 경로(즉시 승인 + 감사 플래그 — D-04)로 사용자가 스스로 구제한다 — 코드로 구제하지 않고, 사람을 기다리게 하지도 않는다(D-00)

- 오류: 해당 없음 — 일일 경계는 판정 규칙이라 별도 실패 모드가 없다. 잘못된 시간대 설정으로 인한 오판정은 SYS-8xx에서 다룬다.
- 복구: 해당 없음 — 리셋할 상태가 없는 lazy 판정 구조(위 두 📌 결정)라 재시작·재진입에 영향받지 않는다.

## device_id (D-07)

#### SYS-301 최초 접속 시 생성·영속 (정상)
- Given: 이 브라우저로 처음 접속한 사용자
- When: 프론트 앱이 부트스트랩된다
- Then: localStorage `stop-island:device-id`가 없으므로 `crypto.randomUUID()`로 생성해 저장한다
- And: 이후 모든 API 호출에 `X-Device-Id` 헤더로 실려 나간다
- And: 새로고침·재방문·탭 여러 개에서도 같은 값이 유지된다 (서버에는 별도 등록 절차 없음 — 첫 verification INSERT 때 자연스럽게 기록)

#### SYS-302 X-Device-Id 헤더 누락 (오류)
- Given: 헤더 없이 API를 직접 호출한다 (curl, 스크립트, localStorage 접근이 막힌 특수 브라우저)
- When: device_id가 필요한 API(`/api/verify`, `/api/reserve`, `/api/guestbook` POST 등)에 요청한다
- Then: 400 + `{ error: { code: "MISSING_DEVICE_ID" } }` — 서버는 익명 요청을 처리하지 않는다
- And: 프론트가 정상 경로라면 이 코드를 받을 일이 없다. 받았다면 localStorage 재생성 후 1회 자동 재시도, 그래도 실패하면 "브라우저 설정을 확인해주세요" 안내

#### SYS-303 localStorage 삭제 / 시크릿 모드 = 새 사용자 (경계 — 한계 명시)
- Given: 오늘 이미 approved 인증을 받은 사용자
- When: localStorage를 지우거나 시크릿 창으로 다시 접속한다
- Then: 새 UUID가 생성되고 서버는 이를 **다른 사용자로 취급한다** — 1일 1회 제한(D-06)이 초기화된다
- And: 이것은 D-07이 명시적으로 수용한 한계다. 로그인 없는 익명 식별의 구조적 상한이며, 실질 방어는 device 단이 아니라 **영수증 단**(DuplicateVerifier — 같은 영수증 재사용 불가)에 있다 (SYS-501)

#### SYS-304 같은 device_id의 동시 요청 (동시성)
- Given: 같은 폰에서 탭 2개가 열려 있다 (같은 device_id)
- When: 두 탭이 동시에 API를 호출한다
- Then: device_id 자체는 충돌 개념이 없다 — 동시 인증 제출의 경합은 SYS-403에서 서버 제약으로 방어한다

#### SYS-305 재방문 시 상태 복원 (복구)
- Given: 인증·예약을 마친 사용자가 브라우저를 완전히 껐다가 몇 시간 뒤 재방문한다
- When: 앱이 부트스트랩된다
- Then: 같은 device_id로 `GET /api/verify/status`를 조회해 당일 인증 상태를 복원하고(D-14 가드 판단), localStorage의 reservation_id로 내 예약을 복원한다 (D-10)
- And: 서버 상태 변화 없음 — device_id가 세션 없는 복원의 열쇠 역할을 한다

## 동시성

> 📌 결정: **"1일 approved 1건"(D-06)은 애플리케이션 체크가 아니라 DB 제약으로 방어한다.** SQLite 부분 UNIQUE 인덱스: `(device_id, verified_kst_date) WHERE status='approved'`. pending 상태가 없으므로(D-02·D-05) pending용 유니크·선삽입 관문도 없다 — 동시 제출은 각자 판정을 끝까지 진행하고, **승인 시점의 approved INSERT가 원자적 관문**이다(둘 다 승인이면 한쪽만 INSERT 성공, 다른 쪽은 유니크 위반 캐치 → 409). 체크 후 INSERT 방식은 요청 2개가 체크를 동시에 통과하는 race가 있다 — design-receipt-ocr.md의 check_and_mark와 같은 원리로, 존재 확인과 기록을 원자화한다. 동시 요청이 둘 다 Gemini 판정까지 진행하는 낭비는 수용한다(같은 이미지면 해시 캐시가 상쇄, 03 VF-901 📌).

#### SYS-401 같은 좌석 동시 예약 — 둘 중 하나만 성공 (동시성/정상)
- Given: 사용자 갑·을이 각자 좌석 화면에서 빈자리 A1을 보고 있다 (30초 폴링 사이라 둘 다 빈자리로 보임)
- When: 둘이 거의 동시에 `POST /api/reserve`로 A1을 예약한다
- Then: 서버는 단일 트랜잭션 안에서 "A1의 active 예약 부재 확인 + INSERT"를 수행 — SQLite 단일 writer 직렬화로 먼저 도착한 쪽만 성공한다
- And: 성공한 갑은 `{ reservation_id, seat, expires_at }`을 받고 예약 완료 화면으로 간다
- And: 을은 409 + `{ error: { code: "SEAT_TAKEN" } }` — 좌석 화면이 즉시 갱신되며 "방금 다른 분이 예약했어요. 다른 자리를 골라주세요" 표시 (D-12)
- And: DB에는 A1의 active 예약이 정확히 1건만 존재한다

#### SYS-402 같은 영수증 동시 제출 — check_and_mark (동시성)
- Given: 같은 영수증 사진이 두 요청으로 동시에 제출된다 (두 기기로 나눠 제출, 또는 더블 탭)
- When: 두 요청이 각각 OCR을 거쳐 DuplicateVerifier에 도달한다
- Then: `DuplicateStore.check_and_mark(key)`(key: 승인번호 우선, 없으면 이미지 해시)가 SQLite UNIQUE 제약 + INSERT 실패 캐치로 **원자적으로** 판정 — 정확히 한 요청만 True(신규)를 받는다
- And: 신규 쪽은 체인을 계속 진행하고, 중복 쪽은 rejected + "이미 사용된 영수증이에요"를 받는다
- And: exists() 후 save() 두 번 호출 구현은 금지 — 둘 다 통과하는 race가 생긴다 (design-receipt-ocr.md 명세)

#### SYS-403 같은 device의 동시 인증 제출 (동시성)
- Given: 같은 폰의 탭 2개에서 서로 다른 영수증으로 동시에 인증을 제출한다
- When: 두 요청이 각각 판정을 끝까지 진행하고(선삽입 관문 없음 — 위 📌) 거의 동시에 승인 트랜잭션에 도달한다
- Then: `(device_id, 당일) WHERE status='approved'` 부분 UNIQUE에 의해 한 건만 INSERT 성공, 나머지는 유니크 위반 캐치 → 409 + `{ error: { code: "ALREADY_VERIFIED_TODAY" } }` (D-06·D-24)
- And: 진 쪽 화면은 `GET /api/verify/status` 재조회로 이긴 쪽의 승인 상태에 동기화된다 (03 VF-901). review성 판정이 섞여 있어도 경로는 같다 — review는 승인+감사 플래그로 승격될 뿐(D-05) 별도 상태·별도 관문이 없다

#### SYS-404 동시 요청량 상한 (경계)
- Given: 좌석 6개·현장 QR 유입 규모에서 순간 동시 요청은 수십 건 수준이다
- When: 피크 트래픽이 들어온다
- Then: SQLite 단일 writer + FastAPI 단일 프로세스로 충분히 소화한다 — 커넥션 풀·샤딩 등은 도입하지 않는다 (과설계). Gemini 동시 호출만 `max_concurrency=5` 세마포어로 제한 (design-receipt-ocr.md)

- 오류: 해당 없음 — 이 절의 409들이 곧 "동시성의 정상적 실패 응답"이다. 별도 오류 모드 없음.
- 복구: 해당 없음 — 모든 방어가 DB 제약·트랜잭션 기반이라 프로세스 재시작과 무관하게 유지된다 (인메모리 락을 쓰지 않는 이유이기도 하다).

## 어뷰징 시나리오 — 방어 수준과 수용 리스크

> 원칙: **7일 팝업, 좌석 6개, 무료 서비스, 무인 운영(D-00).** 어뷰징으로 얻는 이득이 "돗자리 좌석 이용"뿐이므로, 방어 비용 > 피해 비용이 되는 지점부터는 방어하지 않고 리스크를 명시적으로 수용한다. 관용 승인 정책(D-04·D-05)은 어뷰징 표면을 의도적으로 **넓힌** 것이다 — 사용자를 세워두지 않는 대가로 manual 허위 선택·경계선 영수증이 일단 통과한다. 그 안전망이 **원격 사후감사(D-22)**다: 승인은 즉시, 정리는 나중에(무효화 + 예약 해제). 최후의 방어선은 현장 상주 인력이 아니라 감사 큐다.

#### SYS-501 localStorage 삭제로 1일 1회 우회 (수용)
- Given: 오늘 이미 인증한 사용자가 시크릿 창으로 재접속했다 (SYS-303)
- When: 두 번째 인증을 시도한다
- Then: **같은 영수증이면 실패한다** — DuplicateVerifier가 승인번호/이미지 해시로 reject (device와 무관한 방어)
- And: **새 영수증이면 성공한다** — 이건 방어하지 않는다. 우회하려면 실제로 한 번 더 결제해야 하는데, 그것은 어뷰징이 아니라 상권 매출이다 (프로젝트 목적에 부합)
- And: 수용 리스크: 결제 없이 남의 새 영수증을 계속 조달하는 조직적 우회 — 7일 팝업에서 현실성 없음, 수용

#### SYS-502 타인 영수증 사진 재사용 (부분 방어)
- Given: 다른 사람이 이미 인증에 사용한 영수증 사진을 입수했다 (단톡방 공유 등)
- When: 그 사진 또는 재촬영본을 제출한다
- Then: 원본과 같은 이미지는 해시로, 재촬영본은 승인번호로 DuplicateVerifier가 reject — "이미 사용된 영수증이에요"
- And: **아직 아무도 제출 안 한 타인의 영수증은 통과한다** — 영수증의 실소유 검증은 불가능하다. 수용 리스크: "옆 테이블 영수증 주워서 인증"은 막을 수 없음. 어차피 그 영수증도 참여 상점의 당일 실결제이므로 피해는 "결제 안 한 1인의 좌석 이용"에 그친다 — 수용
- And: 승인번호 추출이 None인 영수증의 재촬영본도 통과할 수 있다(해시가 다르므로) — 동일하게 수용. 이런 통과 건 중 review성 판정이 붙은 것은 `needs_audit`로 감사 큐에 남는다(D-05)

#### SYS-503 방명록 도배 (부분 방어 — D-19)
- Given: 익명·무인증 방명록 (D-19)
- When: 같은 device가 1분 안에 두 번째 글을 제출한다
- Then: 429 + `{ error: { code: "RATE_LIMITED" } }` — "잠시 후 다시 써주세요"
- And: rate limit 카운터는 인메모리 — 서버 재시작 시 리셋 수용 (SYS-703)
- And: 수용 리스크: 시크릿 창 갈이나 다중 기기로 1분 1회를 넘는 도배는 가능하다. 방어는 관리자 삭제(ADM-501)로 후행 대응 — 익명 게시판에서 선행 차단을 더 쌓는 건(IP 제한, 캡차) 정상 사용자 UX 훼손 대비 실익이 없다

#### SYS-504 API 직접 호출 (경계)
- Given: 개발자 도구로 API를 직접 호출하는 사용자
- When: 프론트 검증을 건너뛰고 요청을 보낸다
- Then: 모든 규칙(1일 1회, 토큰 검증, 좌석 트랜잭션, rate limit, device 헤더 400)은 **서버에서 재검증**된다 — 프론트 검증은 UX용이지 방어선이 아니다
- And: 임의 device_id를 지어내는 것은 SYS-501과 동일 취급 (새 사용자 = 새 영수증 필요)

#### SYS-505 manual 허위 선택 — 관용 승인의 표면과 사후감사 안전망 (부분 방어)
- Given: 결제하지 않은 사용자가 영수증 없이 상점 직접 선택(manual)으로 제출한다
- When: 서버가 처리한다
- Then: **즉시 승인된다** (D-04 — 무인 운영에서 현장 확인은 불가능, 사용자를 세워둘 수 없음). 단 row에 `needs_audit=1` + `MANUAL_SHOP_SELECTED`가 남아 전건이 감사 큐에 들어간다
- And: 원격 운영진이 감사에서 어뷰징으로 판정하면 인증 무효화(`REVOKED_BY_AUDIT`) + active 예약 해제로 되돌린다 (ADM-212, 03 VF-504) — 어뷰저가 얻는 것은 "감사 전까지의 좌석 이용"뿐
- And: 수용 리스크: 감사가 늦거나 판단 근거가 없으면 무임 이용이 성립한다 — 무료 좌석 7일 팝업에서 피해는 좌석 점유 최대 2시간이며, 관용 정책(D-04)이 명시적으로 수용한 비용이다. 반복 어뷰징(같은 device의 매일 manual)은 감사 큐에서 패턴으로 드러난다

- 정상/오류/동시성/복구: 이 절은 관점 매트릭스가 아니라 위협 모델 목록이다 — 각 시나리오가 "공격(When) → 방어 결과(Then) → 수용선(And)" 구조로 관점을 대체한다.

## 데이터 수명 (D-23)

> 📌 결정: **이미지 삭제 잡 — 매일 04:00 KST, 프로세스 내 일일 태스크.** `팝업 종료일(10/6) + 7일 = 10/13` 이후 첫 실행에서 이미지 디렉토리 전체 삭제 + 해당 verifications의 `image_url`을 NULL로 UPDATE(끊긴 참조를 남기지 않기 위해). 그 전에는 no-op — "업로드 후 7일"의 개별 파일 단위 롤링 삭제는 안 한다. 팝업 7일 + 유예 7일이면 어차피 전체 보존 기간이 최대 2주라, 개별 추적의 복잡도가 실익이 없다.

> 📌 결정: **팝업 종료 후 DB는 삭제하지 않는다.** 방명록·통계는 수업 결과보고와 회고의 핵심 산출물. 개인정보성 데이터는 이미지(카드번호 일부 등)에만 있고 DB에는 익명 device UUID뿐이므로, 이미지만 지우면 보존 리스크가 없다.

#### SYS-601 삭제 기준일 이후 첫 잡 실행 (정상)
- Given: 오늘이 10/13 04:00 KST를 지난 첫 잡 실행 시점이고, 이미지 파일 200개가 저장돼 있다
- When: 삭제 잡이 실행된다
- Then: 이미지 디렉토리의 파일이 전부 삭제되고, `verifications.image_url`이 일괄 NULL로 UPDATE된다
- And: 관리자 화면에서 과거 건을 열어도 이미지 자리는 "보존 기간이 지나 삭제됐어요"로 표시된다 (ADM-204의 플레이스홀더 재사용)

#### SYS-602 기준일 이전의 잡 실행 (경계)
- Given: 팝업 운영 중(10/1) 04:00
- When: 잡이 실행된다
- Then: 기준일(10/13) 전이므로 no-op — 파일·DB 변화 없음. 로그만 남긴다

#### SYS-603 삭제 잡 중 일부 파일 삭제 실패 (오류/복구)
- Given: 파일 하나가 잠겨 있거나 이미 없어 unlink가 실패한다
- When: 잡이 그 파일을 만난다
- Then: 해당 파일만 건너뛰고 나머지는 계속 삭제한다 (전체 롤백 없음), 실패 목록을 로그로 남긴다
- And: 잡은 멱등이므로 다음 날 04:00 실행이 남은 파일을 재시도한다 — "이미 없는 파일"은 성공으로 간주
- And: image_url NULL UPDATE는 파일 삭제 성공 여부와 무관하게 기준일 이후 항상 실행 (참조가 먼저 끊겨도 무해)

- 동시성: 해당 없음 — 기준일 이후에는 신규 업로드가 없고(팝업 종료), 잡은 하루 1회 단일 태스크라 경합 대상이 없다.

## 인프라 복구

#### SYS-701 서버 재시작 — 영속 상태 (정상)
- Given: 배포 또는 장애로 FastAPI 컨테이너가 재시작된다
- When: 앱이 다시 기동한다
- Then: SQLite 파일은 디스크 볼륨에 영속 — verifications(토큰 포함), reservations, guestbook, shops, seats 전부 그대로다. 사용자는 재로그인·재인증 없이 이어서 쓴다 (식별이 localStorage device_id 기반이라 서버 세션이 애초에 없음)
- And: 60초 스윕·일일 삭제 잡 태스크가 기동 시 재시작된다 (SYS-105)

#### SYS-702 서버 재시작 — 인메모리 소실 영향 분석 (경계)
- Given: 재시작으로 프로세스 메모리가 초기화됐다
- When: 서비스가 재개된다
- Then: 소실 항목별 영향은 다음과 같고, 전부 수용한다:
  - **OCR ResultCache(InMemory)**: 같은 이미지 재제출 시 Gemini를 다시 호출 — 비용 몇 원 수준의 낭비뿐, 정합성 무관 (검증 체인은 어차피 캐시 히트에도 재실행)
  - **방명록 rate limit 카운터**: 리셋 — 재시작 직후 1분 창이 한 번 열리는 정도 (SYS-503)
  - **관리자 로그인 실패 카운터**: 리셋 — 무차별 대입 방어 창이 한 번 열림 (ADM 📌 결정에서 수용)
  - **관리자 세션 저장소(인메모리)**: 운영진 재로그인 필요 — 비밀번호 하나 재입력, 허용
- And: **DuplicateStore는 인메모리가 아니라 SQLite다** — 중복 영수증 방어(SYS-402)는 재시작에도 유지된다. 이것만은 소실되면 안 되는 상태라 처음부터 DB에 둔다

#### SYS-703 디스크 이미지 저장 실패 (오류)
- Given: 디스크 풀 또는 권한 문제로 업로드 이미지 쓰기가 실패한다
- When: 사용자가 `POST /api/verify`로 사진을 제출한다
- Then: 서버는 verification 행을 만들지 않고 500 + `{ error: { code: "UPLOAD_FAILED" } }`를 반환한다 — 이미지 저장은 OCR 호출보다 먼저 수행하고, 실패 시 반쪽 상태(행은 있는데 이미지 없음)를 남기지 않는다
- And: 사용자 화면은 "사진 업로드에 실패했어요. 다시 시도해주세요" + 재시도 버튼. 1일 1회 카운트도 소모되지 않는다 (행이 없으므로)
- And: 연속 실패 시 사용자는 manual 경로(상점 직접 선택)로 우회 가능 — 이미지 저장이 없는 경로라 디스크 장애와 무관하게 동작한다

#### SYS-704 Gemini 장애 — manual 폴백 (오류, D-25)
- Given: Gemini가 타임아웃·5xx·rate limit으로 내부 재시도(지수 백오프, max_retries=3)까지 소진하며 실패한다
- When: 사용자의 사진 인증 요청이 이 실패를 만난다
- Then: 서버가 `{ error: { code: "OCR_UNAVAILABLE" } }`를 반환하고, 화면은 "사진 인증이 지금 어려워요. 아래에서 상점을 직접 선택해주세요"와 함께 manual 경로 버튼을 노출한다 (D-25)
- And: manual 제출은 즉시 approved + `needs_audit` 플래그로 저장되어(D-04) 사용자는 곧바로 좌석 선택으로 진행하고, 운영진은 감사 큐(ADM-2xx)에서 사후 확인한다 — 외부 API 장애가 서비스를 전면 중단시키지도, 사용자를 세워두지도 않는다
- And: verification 행은 실패한 photo 시도에 대해 생성되지 않는다 — 재시도·경로 전환이 자유롭다

#### SYS-705 재시작 중 유입된 요청 (동시성/복구)
- Given: 컨테이너 재시작(수 초)의 다운타임 창
- When: 사용자 요청이 이 창에 도달한다
- Then: nginx가 502를 반환하고, 프론트의 폴링(사용자 30초 / 관리자 30·60초)은 다음 주기에 자연 회복한다 — 폴링 기반 설계의 부수 이점으로, 별도 재연결 로직이 필요 없다
- And: 제출성 요청(인증·예약)이 502를 받으면 사용자 재시도로 해결 — 멱등 방어(UNIQUE 제약, check_and_mark)가 중복 반영을 막는다

## 시간대 (D-01)

> 📌 결정: **코드는 컨테이너 TZ 환경변수에 의존하지 않는다.** 모든 "당일" 판정은 명시적으로 `ZoneInfo("Asia/Seoul")`로 계산하고, DB 저장은 UTC로 통일한다. `datetime.now()`(naive) 사용 금지 — 컨테이너 기본 TZ(UTC)와 KST가 섞이면 자정 경계가 9시간 틀어진다. 방어는 설정이 아니라 코드 규약으로 한다: OCI 컨테이너 TZ가 무엇이든 동작이 같아야 한다.

#### SYS-801 기동 시 시간대 자가 점검 (정상)
- Given: 컨테이너가 기동한다
- When: 앱 startup 훅이 실행된다
- Then: `UTC now / KST now / KST 오늘 날짜 / 다음 KST 자정까지 남은 시간`을 로그로 출력한다 — 배포 직후 운영자가 로그 한 줄로 시간대 정상 여부를 확인할 수 있다
- And: 상태 변화 없음 (로그만)

#### SYS-802 UTC 컨테이너에서 자정 경계 판정 (경계 — 검증 시나리오)
- Given: 컨테이너 TZ가 UTC로 설정돼 있다 (OCI 기본값)
- When: KST 00:30(= UTC 15:30, 전날)에 토큰 유효성·today_visitors·DateVerifier 판정이 실행된다
- Then: 모든 판정이 KST 기준 "새 날"로 일관되게 동작한다 — 토큰 만료(SYS-201), 방문자 0 리셋(SYS-204), 당일 영수증 기준이 전부 KST 자정에서 함께 넘어간다
- And: 이 시나리오는 QA 단계에서 TZ=UTC 컨테이너 + 시각 mock으로 자동 테스트한다 — "로컬 개발기(KST)에서는 되는데 서버(UTC)에서 틀어지는" 사고를 배포 전에 잡는 것이 목적

#### SYS-803 서버 시계 오차 (오류 — 수용선)
- Given: OCI 인스턴스의 NTP 동기화가 정상이라는 전제
- When: 시계가 수 초 이내로 어긋난다
- Then: 만료·자정 판정이 그 오차만큼 흔들릴 수 있으나, 모든 판정이 **서버 단일 시계** 기준이므로 판정 간 모순은 생기지 않는다 (클라이언트 시계는 어디에도 쓰지 않음 — D-01)
- And: 분 단위 이상의 시계 이탈 감지는 하지 않는다 — NTP가 도는 관리형 인스턴스에서 과설계. 수용

- 동시성: 해당 없음 — 시간대는 판정 규칙이지 경합 자원이 아니다. 자정 순간의 경합은 SYS-205에서 다뤘다.
- 복구: 해당 없음 — 시간대 계산은 무상태(코드 상수 + 시스템 시계)라 재시작의 영향이 없다.
