# 06 — 관리자 시나리오 (ADM)

> **한 줄 요약:** 무인 팝업(D-00)을 **원격에서 모니터링·정리하는 도구** — 로그인, needs_audit 인증 사후감사(문제없음/어뷰징 무효화), 좌석·예약·방명록·상점 관리, 당일 통계 (D-21, D-22 기준). 사용자를 기다리게 하는 승인/거부 큐는 존재하지 않는다 — 모든 사용자 플로우는 이미 즉시 완결됐고, 관리자는 뒤에서 정리한다.

**커버 범위**
- 로그인 / 세션 (ADM-1xx)
- 감사 큐 — needs_audit 인증 사후 확인 (ADM-2xx)
- 좌석 관리 (ADM-3xx)
- 예약 관리 (ADM-4xx)
- 방명록 관리 (ADM-5xx)
- 상점 관리 (ADM-6xx)
- 당일 통계 대시보드 (ADM-7xx)

> 📌 결정: **관리자 화면 구성 — 모바일 우선 단일 페이지 `/admin`.** 운영진은 현장에 상주하지 않고(D-00) 각자 폰·노트북으로 원격 접속한다. 상단에 5개 탭: `감사` / `좌석·예약` / `방명록` / `상점` / `통계`. 기본 진입 탭은 `통계`(원격 모니터링의 첫 질문은 "지금 현장이 어떤가"). `감사` 탭 라벨에 미감사(needs_audit=1 AND audited_at IS NULL) 건수 배지 표시. 데스크톱은 같은 레이아웃이 가운데 정렬로 뜨는 정도로 충분 — 별도 데스크톱 레이아웃 안 만든다.

> 📌 결정: **로그인 무차별 대입 방어 — IP 기준 rate limit.** 로그인 실패가 같은 IP에서 10분 내 5회 누적되면 이후 시도를 5분간 429로 차단. 카운터는 인메모리(dict)로 충분 — 서버 재시작 시 리셋되는 리스크는 7일 팝업 규모에서 수용(07-system SYS-7 참고). 계정 잠금(lockout)은 안 한다 — 비밀번호가 하나뿐이라 잠기면 운영진 전원이 잠긴다.

## 관리자 로그인 / 세션

### 로그인

#### ADM-101 올바른 비밀번호로 로그인 성공 (정상)
- Given: 운영진이 `/admin`에 접속해 로그인 화면을 보고 있다
- When: 올바른 비밀번호를 입력하고 로그인 버튼을 누른다
- Then: 서버가 `POST /api/admin/login`에 200 + HttpOnly 세션 쿠키(만료 12시간, D-21)를 내려준다
- And: 화면이 관리자 메인(`통계` 탭)으로 전환된다
- And: 서버는 세션 토큰을 발급·저장한다 (인메모리 세션 저장소)

#### ADM-102 잘못된 비밀번호 (오류)
- Given: 로그인 화면
- When: 틀린 비밀번호를 입력하고 로그인 버튼을 누른다
- Then: 서버가 401 + `{ error: { code: "INVALID_PASSWORD", message } }`를 반환한다 (D-24)
- And: 화면에 "비밀번호가 올바르지 않아요" 표시, 입력 필드는 유지·포커스
- And: 서버는 해당 IP의 실패 카운터를 +1 한다

#### ADM-103 로그인 실패 5회 누적 — rate limit (경계/오류)
- Given: 같은 IP에서 10분 내 로그인 실패가 4회 누적된 상태
- When: 5번째 로그인 시도가 실패한다
- Then: 이후 해당 IP의 로그인 요청은 5분간 429 + `{ error: { code: "TOO_MANY_ATTEMPTS", message } }`
- And: 화면에 "시도가 너무 많아요. 5분 후 다시 시도해주세요" 표시
- And: 이 5분 동안은 올바른 비밀번호를 넣어도 429 (카운터 우선)
- And: 5분 경과 후 카운터가 초기화되어 정상 로그인 가능

#### ADM-104 세션 만료 후 API 호출 (경계)
- Given: 로그인한 지 12시간이 지나 세션이 만료된 관리자 화면이 열려 있다
- When: 아무 관리자 API(`/api/admin/*`)를 호출한다 (예: 통계 폴링)
- Then: 서버가 401 + `{ error: { code: "SESSION_EXPIRED" } }`를 반환한다
- And: 화면이 로그인 화면으로 전환되고 "세션이 만료됐어요. 다시 로그인해주세요" 표시
- And: 서버는 만료된 세션을 저장소에서 제거한다

#### ADM-105 비로그인 상태로 /admin 접근 (오류)
- Given: 세션 쿠키가 없는 브라우저
- When: `/admin` 라우트에 직접 접근한다 (URL 입력, 북마크 등)
- Then: 관리자 데이터는 렌더되지 않고 로그인 화면이 표시된다
- And: 프론트가 최초 세션 확인 API에서 401을 받아 로그인 폼으로 분기한다 — 서버 데이터는 일절 내려가지 않는다

#### ADM-106 두 운영진이 각자 기기로 동시에 로그인 (동시성)
- Given: 운영진 2명이 각자의 폰·노트북에서 로그인 화면을 보고 있다
- When: 둘 다 올바른 비밀번호로 거의 동시에 로그인한다
- Then: 두 요청 모두 200, 서로 다른 세션 토큰이 발급된다
- And: 세션은 기기별 독립 — 한쪽 로그인이 다른 쪽을 무효화하지 않는다 (동시 세션 수 제한 없음)

#### ADM-107 로그인 상태에서 새로고침 (복구)
- Given: 로그인해서 관리자 화면을 쓰고 있다
- When: 브라우저를 새로고침하거나 탭을 닫았다가 다시 연다
- Then: HttpOnly 쿠키가 살아 있으므로 재로그인 없이 관리자 화면이 다시 렌더된다
- And: 서버 상태 변화 없음

## 감사 큐 (D-22)

사용자는 이미 승인되어 진행 중이다(D-04·D-05). 감사는 **사후 절차**라 실시간성이 필요 없다 — 늦게 봐도 사용자를 막지 않고, 어뷰징이면 그때 되돌린다(무효화 + 예약 해제).

> 📌 결정: **감사 큐 갱신은 60초 폴링 + 수동 새로고침 버튼.** 사용자를 기다리게 하던 승인 큐(구 10초 폴링)와 달리 감사는 지연 비용이 "어뷰저가 좌석을 몇 분 더 쓰는 것"뿐이다. 탭이 백그라운드면 폴링 중단(visibilitychange), 복귀 시 즉시 1회 갱신. "N초 내 감지" 같은 지연 목표는 두지 않는다.

> 📌 결정: **감사 판정은 2택 — [문제없음] / [어뷰징 — 무효화].** 거부 사유 선택지는 없다(사용자에게 전달할 사유가 없음 — 이미 승인된 건을 정리하는 것이므로). ① 문제없음: `needs_audit=0` + `audited_at` 기록, 사용자 영향 없음. ② 어뷰징 — 무효화: 확인 다이얼로그("이 인증을 무효화할까요? 연결된 예약도 해제돼요")를 거쳐, 단일 트랜잭션으로 verification을 `rejected` + `REVOKED_BY_AUDIT` + `audited_at`으로 전환하고 연결된 active 예약을 `cancelled`로 해제한다(03 VF-504). 판단이 애매하면 **문제없음으로 처리한다** — 이미 통과시킨 관용 정책(D-05)의 연장선이며, 무효화는 명백한 어뷰징에만 쓴다.

### 큐 목록

#### ADM-201 감사 대기 목록 표시 (정상)
- Given: 미감사(needs_audit=1, audited_at IS NULL) 인증이 3건 있는 상태로 관리자가 `감사` 탭에 있다
- When: 탭이 로드된다 (`GET /api/admin/verifications?audit=pending`)
- Then: 3건이 오래된 순으로 카드 목록으로 표시된다
- And: 각 카드에는 ① 영수증 이미지 원본(photo인 경우, 탭하면 확대) ② OCR 추출 결과(상호명·결제일·금액·confidence) ③ 플래그 사유(예: "confidence 0.45 < 0.6" `LOW_CONFIDENCE`, "상점 fuzzy 0.62 경계" `SHOP_MATCH_UNCERTAIN`, "상점 직접 선택" `MANUAL_SHOP_SELECTED`) ④ method 뱃지(`photo`/`manual`) ⑤ 제출 시각 ⑥ **연결된 예약 상태**(active면 좌석 라벨, 없으면 "예약 없음")가 표시된다
- And: manual 건은 이미지 대신 "상점 직접 선택: {상점명}" 표시 (D-04)
- And: 서버 상태 변화 없음 (조회만)

#### ADM-202 감사 대기 0건 (경계)
- Given: 미감사 건이 하나도 없다
- When: `감사` 탭이 로드된다
- Then: "감사할 인증이 없어요" 빈 상태 표시, 탭 배지 숨김
- And: 60초 폴링은 계속 돈다 (새 건 도착 감지)

#### ADM-203 새 감사 건 도착 (정상)
- Given: 관리자가 `감사` 탭을 열어두고 있다
- When: 60초 폴링 또는 수동 새로고침에서 새 needs_audit 건이 응답에 포함된다
- Then: 목록 하단에 새 카드가 추가되고 탭 배지 숫자가 갱신된다
- And: 관리자가 보던 카드의 스크롤 위치는 유지된다 (목록 전체 리렌더로 위치를 잃지 않는다)

#### ADM-204 영수증 이미지 로드 실패 (오류)
- Given: 감사 카드의 image_url이 깨져 있다 (파일 삭제·디스크 문제)
- When: 카드가 렌더된다
- Then: 이미지 자리에 "이미지를 불러올 수 없어요" 플레이스홀더 표시
- And: OCR 추출 결과·판정 버튼은 정상 동작한다. 이미지 없이 판단이 안 서면 **문제없음으로 처리**한다(위 📌 — 이미 승인된 건이고, 무효화는 명백한 근거가 있을 때만)

### 감사 판정

#### ADM-211 문제없음 처리 (정상)
- Given: 감사 카드 1건 (LOW_CONFIDENCE, 영수증이 정상으로 보임)
- When: 관리자가 [문제없음]을 누른다 (`PATCH /api/admin/verifications/:id` → `audit: "ok"`)
- Then: 서버가 `needs_audit=0`, `audited_at` 기록 — status·token·예약은 건드리지 않는다
- And: 관리자 화면에서 해당 카드가 목록에서 사라지고 배지 -1
- And: **사용자에게는 아무 일도 일어나지 않는다** — 처음부터 끝까지 통상 승인과 무차이 (D-05, 03 VF-503)

#### ADM-212 어뷰징 판정 — 무효화 + 예약 해제 (정상)
- Given: 감사 카드 1건 (manual 제출인데 해당 시간대 상점 결제 내역이 없다는 상점 측 확인 등 명백한 어뷰징), 해당 사용자는 active 예약 보유 중
- When: 관리자가 [어뷰징 — 무효화]를 누르고 확인 다이얼로그를 확정한다 (`PATCH /api/admin/verifications/:id` → `audit: "revoke"`)
- Then: 서버가 단일 트랜잭션으로 ① verification을 `rejected` + `REVOKED_BY_AUDIT` + `audited_at`으로 전환(토큰 무효) ② 연결된 active 예약을 `cancelled`로 해제 (03 VF-504)
- And: 카드가 목록에서 사라지고, 좌석은 즉시 빈자리로 계산된다
- And: 사용자는 다음 조회 때 알게 된다 — 완료 화면 재조회 시 "예약이 취소되었어요"(04 RSV-608), `/verify` 재진입 시 재인증 안내(03 VF-505). 정당한 영수증이 있으면 재인증 가능

#### ADM-213 두 운영진이 같은 건을 동시에 감사 (동시성)
- Given: 운영진 A, B가 각자 기기에서 같은 감사 카드를 보고 있다
- When: A가 문제없음, B가 무효화를 거의 동시에 누른다
- Then: 서버는 `needs_audit = 1 AND audited_at IS NULL` 조건부 UPDATE로 처리 — 먼저 도착한 요청만 성공(200)
- And: 나중 요청은 409 + `{ error: { code: "ALREADY_PROCESSED" } }`
- And: 늦은 쪽 화면에 "다른 운영진이 이미 처리했어요" 토스트 + 목록 즉시 갱신
- And: 최종 상태는 먼저 처리된 판정 하나만 반영된다 (문제없음↔무효화 뒤집기 없음)

#### ADM-214 이미 감사된 건 재처리 시도 (오류)
- Given: 폴링 갱신 전이라 관리자 화면에 이미 처리된 건의 카드가 남아 있다
- When: 문제없음 또는 무효화 버튼을 누른다
- Then: 409 `ALREADY_PROCESSED` — ADM-213과 동일한 방어가 그대로 동작한다
- And: 화면에서 카드 제거 + 목록 갱신. 서버 상태는 변하지 않는다

#### ADM-215 감사 요청 중 네트워크 단절 (복구)
- Given: 관리자가 [문제없음]을 눌렀는데 응답을 못 받았다
- When: 관리자가 같은 카드에서 다시 누른다
- Then: 첫 요청이 서버에 도달했었다면 409 `ALREADY_PROCESSED` — 프론트는 이 코드를 "성공한 것"으로 간주하고 카드를 제거한다 (판정이 이미 반영됐다는 뜻이므로)
- And: 첫 요청이 도달하지 않았었다면 이번 요청이 정상 처리(200)된다
- And: 어느 쪽이든 최종 서버 상태는 판정 1회 처리로 동일 (멱등)

## 좌석 관리

> 📌 결정: **active 예약이 있는 좌석을 닫으면 예약은 유지된다.** `is_open = false`는 "신규 예약 차단"만 의미. 이미 앉아 있는 사람을 시스템이 쫓아내는 건 갈등 소지 — 내보내야 하면 운영진이 예약 수동 해제(ADM-4xx)를 별도로 실행한다. 닫기 시도 시 active 예약이 있으면 확인 다이얼로그에 "이 좌석에 진행 중인 예약이 있어요. 예약은 유지되고 새 예약만 막혀요"를 표시한다.

> 📌 결정: **예약 이력이 있는 좌석은 삭제 불가.** `reservations.seat_id` FK가 걸려 있으므로, 이력이 1건이라도 있으면 삭제 대신 "닫기"를 안내한다(409). 삭제는 팝업 초기 세팅 단계(이력 0건)에서만 실질적으로 쓰인다.

### 열기 / 닫기

#### ADM-301 빈 좌석 닫기 (정상)
- Given: A1 좌석이 열려 있고 active 예약이 없다 (예: 원격 CCTV·현장 연락으로 파라솔 파손을 확인했다)
- When: 관리자가 A1의 닫기 토글을 누르고 확인한다 (`PATCH /api/admin/seats/a1` → `is_open: false`)
- Then: 서버가 `seats.is_open = false`로 변경
- And: 관리자 화면에서 A1이 "닫힘" 상태로 표시된다
- And: 이후 사용자 좌석 화면(04)의 30초 폴링에서 A1이 선택 불가로 바뀌고, `/api/status`의 빈자리 수에서 제외된다

#### ADM-302 active 예약이 있는 좌석 닫기 (경계)
- Given: A1에 active 예약이 있다
- When: 관리자가 A1 닫기를 누른다
- Then: "진행 중인 예약이 있어요. 예약은 유지되고 새 예약만 막혀요" 확인 다이얼로그 표시
- And: 확인하면 `is_open = false`, 기존 예약의 status·expires_at은 변하지 않는다
- And: 예약자의 예약 완료 화면(D-09)도 그대로 유효 — 사용자에게는 아무 변화가 없다
- And: 예약이 만료/해제된 뒤에도 좌석은 닫힌 상태라 신규 예약 불가

#### ADM-303 닫힌 좌석 열기 (정상)
- Given: A1이 닫혀 있다
- When: 관리자가 열기 토글을 누른다
- Then: `is_open = true`. 다음 사용자 폴링부터 A1이 예약 가능으로 표시된다

### 추가 / 삭제

#### ADM-311 좌석 추가 (정상)
- Given: 현장에 평상이 하나 더 들어와 좌석을 늘리려 한다
- When: 관리자가 id `c1`, 라벨 `C1`, 정원 4로 좌석 추가를 실행한다 (`POST /api/admin/seats`)

- And: 사용자 좌석 화면 폴링에 C1이 나타난다

#### ADM-312 중복 id로 좌석 추가 (오류)
- Given: `a1` 좌석이 이미 존재한다
- When: id `a1`로 좌석 추가를 시도한다
- Then: 409 + `{ error: { code: "SEAT_ID_TAKEN" } }`, 화면에 "이미 있는 좌석 ID예요" 표시. DB 변화 없음

#### ADM-313 예약 이력 있는 좌석 삭제 시도 (오류)
- Given: A1에 예약 이력(active든 expired든)이 1건 이상 있다
- When: 관리자가 A1 삭제를 시도한다 (`DELETE /api/admin/seats/a1`)
- Then: 409 + `{ error: { code: "SEAT_HAS_RESERVATIONS" } }`
- And: 화면에 "예약 이력이 있는 좌석은 삭제할 수 없어요. 대신 닫아주세요" 표시. DB 변화 없음

#### ADM-314 이력 없는 좌석 삭제 (경계)
- Given: 방금 잘못 추가한 `c9` 좌석, 예약 이력 0건
- When: 삭제를 확인한다
- Then: seats에서 행이 제거되고 목록·사용자 화면에서 사라진다

- 동시성: 해당 없음 — 좌석 열기/닫기·추가/삭제는 운영진 2명이 같은 좌석을 동시에 조작할 확률이 극히 낮고, 최악의 경우도 마지막 쓰기가 남는(last-write-wins) 토글이라 데이터가 깨지지 않는다. 삭제 경합은 두 번째 요청이 404를 받는 것으로 끝난다.
- 복구: 해당 없음 — 좌석 조작은 단건 멱등 연산(토글·INSERT·DELETE). 네트워크 단절 후 재시도해도 같은 최종 상태에 도달한다.

## 예약 관리

#### ADM-401 active 예약 목록 (정상)
- Given: active 예약이 2건 있다
- When: 관리자가 `좌석·예약` 탭을 연다 (`GET /api/admin/reservations?status=active`)
- Then: 좌석 라벨·예약 시각·만료 예정 시각(expires_at)이 목록으로 표시된다
- And: 서버 상태 변화 없음

#### ADM-402 노쇼/자리 비움 수동 해제 (정상)
- Given: 원격 확인(현장 파트너 연락·CCTV 등) 결과 A1 예약자가 자리를 비운 지 오래라 좌석을 회수하려 한다
- When: 관리자가 해당 예약의 해제 버튼을 누르고 확인한다 (`DELETE /api/admin/reservations/:id`)
- Then: 서버가 reservation을 `cancelled`로 변경한다 (행 삭제 아님 — 이력 보존)
- And: 관리자 목록에서 제거, A1이 즉시 빈자리로 계산된다
- And: 원 예약자의 예약 완료 화면은 다음 재조회(`GET /api/reservations/:id`)에서 cancelled를 받아 "예약이 취소되었어요" 상태로 전환된다 (04 RSV-608과 동일 화면)
- And: 원 예약자의 verify token은 당일 유효하므로 재예약 가능 (D-08) — 인증까지 무효화하는 감사 판정(ADM-212)과 이 점이 다르다

#### ADM-403 이미 만료된 예약을 해제 시도 (경계)
- Given: 폴링 갱신 전 화면에 남아 있던 예약이 그 사이 2시간 만료(D-11)로 expired 처리됐다
- When: 관리자가 해제 버튼을 누른다
- Then: 409 + `{ error: { code: "NOT_ACTIVE" } }` — 화면은 "이미 종료된 예약이에요" 토스트 + 목록 갱신
- And: expired 상태는 그대로 유지된다 (expired → cancelled로 덮어쓰지 않음)

#### ADM-404 해제 직후 그 좌석을 다른 사용자가 예약 (동시성)
- Given: 관리자가 A1 예약을 수동 해제했다
- When: 좌석 화면을 보고 있던 다른 사용자가 폴링 갱신으로 A1이 빈 것을 보고 즉시 예약한다
- Then: A1에 active 예약이 없으므로 예약 트랜잭션이 정상 성공한다 — 해제와 신규 예약 사이에 별도 쿨다운 없음
- And: 만약 해제 처리와 신규 예약 요청이 진짜 동시라면, 예약 INSERT 트랜잭션의 active 중복 체크가 최종 방어한다 (D-12, 07-system SYS-401)

#### ADM-405 해제 요청 중 네트워크 단절 (복구)
- Given: 해제 버튼을 눌렀는데 응답을 못 받았다
- When: 다시 해제를 누른다
- Then: 첫 요청이 반영됐다면 409 `NOT_ACTIVE` — 프론트는 목록에서 제거하고 끝 (결과 동일, 멱등)
- And: 반영 안 됐다면 이번 요청이 정상 처리된다

- 오류: ADM-403이 오류 관점을 겸한다 (비-active 대상 조작 거부).

#### ADM-601 상점 추가 (정상)

- Given: 상점 이름 표기가 잘못 등록돼 있다
- When: 관리자가 이름/카테고리/정렬을 수정한다 (`PATCH /api/admin/shops/:id`)
- Then: shops 행이 갱신되고 사용자 화면·OCR 허용 목록에 다음 요청부터 반영된다
- And: 슬러그(id)는 수정 불가 — 참조 무결성(verifications, shop_tags) 때문. 바꾸려면 새 상점 추가 + 기존 비활성

- When: 관리자가 비활성화한다 (`PATCH /api/admin/shops/:id` → `is_active: false`)

- And: 기존 approved verification 5건과 그 토큰·active 예약은 전혀 영향받지 않는다

#### ADM-605 비활성 직전 제출된 인증 (동시성)
- Given: 사용자가 해당 상점 영수증으로 인증을 제출하는 순간 관리자가 상점을 비활성화했다
- When: 인증 처리 시점의 allowed_stores 조회가 비활성화 이후에 실행된다
- Then: StoreVerifier가 매칭 실패 — 유사도에 따라 `rejected`(명백 불일치) 또는 **승인 + `SHOP_MATCH_UNCERTAIN` 감사 플래그**(경계선, D-05)로 갈린다. 어느 쪽도 사용자를 세워두지 않는다
- And: 감사 플래그로 들어온 건은 운영진이 감사 큐에서 상황(비활성 직전 결제)을 보고 문제없음 처리한다 — 경계 순간의 건은 사람이 사후에 정리하는 것으로 충분, 자동화하지 않는다

- 복구: 해당 없음 — 상점 조작은 단건 멱등 갱신. 재시도 시 같은 최종 상태.
