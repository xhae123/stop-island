# 02 메뉴 선택 — Given/When/Then 시나리오

> **한 줄 요약**: 분기 화면(`/menu`)의 카드 즉시 push 내비게이션(D-16), 뒤로가기, 참여 상점(`GET /api/shops`) 로딩/성공/빈 목록/실패 폴백, 접근성, 직접 URL 진입을 커버한다. 좌석은 이 화면에서 고르지 않는다 — QR로 이미 정해져 있다(D-26).

> 📌 결정(D-26): "테이블 예약" 가지는 **"이 자리(QR로 찍은 자리)를 잡는다"**는 의미다. 좌석 선택은 이 화면에 없고 `stop-island:seat`(QR 부트 시 저장)로 흐름 전체에 전달된다. 메뉴 구조는 그대로(예약/방명록 두 갈림길) — 좌석이 QR로 고정됐을 뿐이다. seat 없이 진입한 경우에도 이 화면은 분기만 하고, 좌석 QR 요구는 예약 경로(RSV-106)로 넘긴다.

**커버 범위**
- 메뉴 카드 탭 즉시 push (D-16) — 라디오 선택 아님
- 뒤로가기(헤더 ←), 직접 URL 진입, 새로고침
- shops API 로딩/성공(배지 N개 + "+n개")/빈 목록/실패 폴백
- 접근성 — 카드 전체 탭 영역
- 인증 완료 상태에서의 진입 (D-14 연계)

---

## 기능 영역: 진입·헤더·복구

### 그룹 1 — 진입 경로와 복귀

#### MENU-101 메인에서 진입 (정상)
- Given: 메인(01)에서 "테이블 예약하기" CTA를 탭했다
- When: `/menu`가 push된다
- Then: 헤더(← + "무엇을 하시겠어요?"), 카드 2개(테이블 예약 / 방명록 작성), 참여 상점 섹션이 렌더되고 `GET /api/shops` 요청이 시작된다
- And: 서버 상태 변화 없음(읽기 전용 화면). localStorage 변화 없음

#### MENU-102 헤더 뒤로가기 → 메인 복귀 (정상)
- Given: `/menu`가 메인에서 push되어 열려 있다
- When: 헤더 ← 버튼(또는 브라우저 뒤로가기)을 탭한다
- Then: 히스토리 back으로 메인(01)에 복귀하고, 메인은 재마운트되어 status 폴링을 재시작한다(MAIN-406)
- And: 서버·localStorage 변화 없음

#### MENU-103 직접 URL 진입 (경계)
- Given: 히스토리 없이 브라우저 주소창에 `/menu`를 직접 입력해 연다 (또는 공유 링크로 진입)
- When: 화면이 로드된다
- Then: 가드 없이 정상 렌더된다. device_id가 없으면 이 시점에 생성·저장한다(MAIN-101 규칙)
- And: 헤더 ← 탭 시 뒤로 갈 히스토리가 없으므로 메인(`/`)으로 replace 이동한다
- And: 서버 상태 변화 없음

> 📌 결정: `/menu`는 라우트 가드를 두지 않는다(인증·예약 전 단계의 순수 분기 화면). 히스토리가 없는 상태에서 ←를 탭하면 메인으로 replace 이동한다 — back이 앱 밖(빈 탭)으로 나가는 것 방지.

#### MENU-104 새로고침 (복구)
- Given: `/menu`가 열려 있고 상점 배지가 표시된 상태다
- When: 브라우저를 새로고침한다
- Then: 화면이 그대로 재렌더되고 `GET /api/shops`를 다시 호출한다(클라이언트 캐시 없음)
- And: 서버·localStorage 변화 없음

- **오류**: 해당 없음 — 진입/복귀 자체는 서버 호출이 없다(shops API 오류는 그룹 3에서 커버).
- **동시성**: 해당 없음 — 진입·복귀는 단일 탭 내 히스토리 조작이며 경합 자원이 없다.

## 기능 영역: 메뉴 카드 내비게이션 (D-16)

### 그룹 2 — 카드 탭 즉시 push

#### MENU-201 "테이블 예약" 카드 탭 (정상)
- Given: `/menu`가 렌더된 상태다
- When: "테이블 예약"(= 이 자리 잡기) 카드(row 어디든)를 탭한다
- Then: selected 스타일(`border: 2px solid #FFEE00`)이 press 피드백으로 순간 표시된 뒤 **즉시** `/verify`(03 영수증 인증)로 push된다 — 라디오 선택 유지·별도 "다음" 버튼 없음. 좌석 선택 단계는 없다(자리는 QR로 고정 — D-26)
- And: 서버·localStorage 변화 없음(목적값을 서버로 전송하지 않음. 좌석은 이미 `stop-island:seat`에 있음)

#### MENU-202 "방명록 작성" 카드 탭 (정상)
- Given: `/menu`가 렌더된 상태다
- When: "방명록 작성" 카드를 탭한다
- Then: press 피드백 후 즉시 `/guestbook`(05 방명록)으로 push된다 — 영수증 인증을 거치지 않는다(방명록은 무인증)
- And: 서버·localStorage 변화 없음

#### MENU-203 빠른 연속 탭 — 중복 push 방지 (동시성)
- Given: `/menu`가 렌더된 상태다
- When: 같은 카드를 짧은 간격으로 2회 이상 탭한다(더블 탭)
- Then: 라우트 push는 1회만 발생한다(히스토리에 대상 화면이 중복으로 쌓이지 않음)
- And: 서버·localStorage 변화 없음

> 📌 결정: 카드 탭 핸들러는 내비게이션 전환이 시작되면 후속 탭을 무시한다(전환 중 플래그). back 시 `/verify`가 두 번 나오는 히스토리 오염 방지.

#### MENU-204 접근성 — 카드 전체 탭 영역 (경계)
- Given: 스크린 리더 또는 키보드로 화면을 탐색한다
- When: 카드에 포커스를 두고 Enter/Space를 누르거나, 카드 row의 가장자리(아이콘·서브텍스트 영역 포함)를 탭한다
- Then: 카드 전체가 단일 button 역할로 동작해 MENU-201/202와 동일하게 push된다
- And: chevron(>) 아이콘은 `aria-hidden`으로 낭독에서 제외되고, 카드에는 "테이블 예약 — 영수증 인증 후 이 자리 예약" 형태의 접근 가능한 레이블이 있다(좌석은 QR로 고정 — D-26)
- And: 서버·localStorage 변화 없음

#### MENU-205 이미 인증 완료(approved) 상태에서 "테이블 예약" 탭 (경계, D-14)
- Given: 오늘 이미 approved 토큰이 localStorage `stop-island:verify`에 있다
- When: "테이블 예약" 카드를 탭한다
- Then: 라우팅은 분기 없이 동일하게 `/verify`로 push되며, 03 화면이 "오늘 인증 완료" 상태로 렌더된다(D-14 — 상태 판단은 03 화면 책임, 이 화면은 분기 로직을 갖지 않음)
- And: 서버·localStorage 변화 없음

- **오류**: 해당 없음 — 카드 탭은 서버 호출이 없는 순수 클라이언트 라우팅이다.
- **복구**: 해당 없음 — 카드에는 유지할 상태가 없다(새로고침 복구는 MENU-104가 커버).

## 기능 영역: 참여 상점 섹션 — `GET /api/shops`

### 그룹 3 — 배지 렌더링과 폴백

> 📌 결정: 배지 노출 규칙 — 응답 목록(서버가 `is_active = true`, `sort_order` 정렬로 반환) 중 **상위 3개**를 배지로 표시하고, 나머지 개수를 "+{n}개" plain text로 붙인다. n = 전체 − 3이며, n ≤ 0이면 "+n개"를 표시하지 않는다. 배지·"+n개"에 탭 인터랙션은 없다(정적 표시).

#### MENU-301 로딩 → 성공 (정상)
- Given: `/menu`에 진입해 `GET /api/shops` 응답 대기 중이다
- When: 응답이 도착한다 (예: 상점 15개)
- Then: 대기 동안 배지 영역에 skeleton placeholder를 표시하다가, 상위 3개 배지("막걸리계보", "조준영 목공방", …) + "+12개" 텍스트로 교체된다
- And: 서버·localStorage 변화 없음

#### MENU-302 상점 3개 이하 (경계)
- Given: 활성 상점이 2개뿐이다
- When: `GET /api/shops` 응답이 도착한다
- Then: 배지 2개만 표시하고 "+n개" 텍스트는 표시하지 않는다
- And: 서버·localStorage 변화 없음

#### MENU-303 상점 0개 — 빈 목록 (경계)
- Given: 활성 상점이 없다(운영 준비 전 또는 전체 비활성)
- When: `GET /api/shops`가 빈 배열을 반환한다
- Then: 섹션 자체는 유지하되 배지 대신 "참여 상점을 준비 중이에요" 문구를 표시한다. 카드 2개 등 나머지 기능은 정상
- And: 서버·localStorage 변화 없음

> 📌 결정: 빈 목록 문구는 "참여 상점을 준비 중이에요". 섹션을 숨기지 않는 이유 — 레이아웃 점프 방지와 "참여 상점이 존재한다"는 컨셉 전달 유지.

#### MENU-304 shops API 실패 (오류)
- Given: `GET /api/shops`가 5xx 또는 네트워크 오류로 실패한다
- When: 실패를 감지한다
- Then: 배지 영역에 폴백 문구 "참여 상점 정보를 불러오지 못했어요"를 표시한다. 화면을 블로킹하지 않으며 카드 2개는 정상 동작한다
- And: 자동 재시도·재시도 버튼은 두지 않는다(재진입/새로고침 시 재조회). 서버·localStorage 변화 없음

> 📌 결정: shops 실패 시 폴백 문구는 "참여 상점 정보를 불러오지 못했어요". 폴링·자동 재시도 없음 — 이 섹션은 참고 정보라 실패해도 핵심 플로우(예약/방명록)에 지장이 없다.

#### MENU-305 응답 도착 전 카드 탭으로 이탈 (동시성)
- Given: `GET /api/shops`가 아직 in-flight다
- When: 사용자가 카드를 탭해 `/verify` 또는 `/guestbook`으로 push된다
- Then: 이동은 API 응답을 기다리지 않고 즉시 일어난다. unmount 이후 도착한 shops 응답은 무시된다(상태 갱신·콘솔 에러 없음)
- And: 서버·localStorage 변화 없음

#### MENU-306 실패 후 재진입 시 재조회 (복구)
- Given: shops 로딩에 실패해 폴백 문구가 표시된 상태에서 다른 화면으로 이동했다
- When: back 또는 재진입으로 `/menu`에 돌아온다
- Then: 재마운트 시 `GET /api/shops`를 다시 호출하고, 성공하면 배지가 정상 표시된다(실패 상태를 기억하지 않음)
- And: 서버·localStorage 변화 없음
