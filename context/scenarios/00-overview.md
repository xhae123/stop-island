# 시나리오 개요 — Given/When/Then 전체 커버리지

> 이 폴더는 "멈춰, 섬!" 웹 서비스의 **기획 초안을 겸하는 시나리오 명세**다.
> 와이어프레임 스펙(`../wireframe-spec/`)이 비워둔 구멍을 전부 결정으로 메웠다.
> 아래 **결정표**가 이 폴더 전체의 단일 진실이다. 각 시나리오 파일은 결정표를 전제로 쓴다.

## 시나리오 작성 규약

- 계층: `## 기능 영역` → `### 시나리오 그룹` → `#### {ID} 시나리오 제목` → Given/When/Then 불릿.
- ID: `{접두어}-{그룹번호}{연번}` (예: `VF-201`). 이슈/테스트 코드에서 이 ID로 참조한다.
- 각 기능은 최소 5개 관점을 커버한다: **정상(happy) / 경계(boundary) / 오류(error) / 동시성(race) / 복구(recovery — 새로고침·재진입·네트워크 단절 후)**. 해당 없는 관점은 "해당 없음 — 이유"를 명시한다.
- Then에는 화면 표시 + 서버 상태 변화 + 저장소(localStorage 등) 변화를 모두 적는다.
- 이 문서가 새로 내린 결정은 각 파일에서 `> 📌 결정:` 인용 블록으로 표시한다. Tom이 뒤집으면 해당 블록만 고치면 된다.

## 플로우 전체도 (결정 반영 — D-26 좌석별 QR)

```
좌석 A3의 QR 스캔(/#/?seat=a3) → 01 메인 (자리 맥락: "A3에서 시작") ─ 햄버거 메뉴(01/02/05)
             │  [내 예약 보기 배너 — active 예약 보유 시]
             │  A3가 taken/closed면 → "사용 중" + 현황 + "빈 자리 QR 찍어주세요"(예약 CTA off)
             ├─ [이 자리(A3) 잡기] → 02 메뉴 선택 ─ 테이블 예약 → 03 영수증 인증
             │                        └ 방명록 작성 → 05 방명록
             │        03: 사진 업로드 → OCR
             │            ├ approved → 04 이 자리 확정 (애매한 건은 승인+감사 플래그 — D-05)
             │            ├ retry/rejected → 03 재시도 (사유 표시)
             │            ├ manual(상점 직접 선택) → 즉시 승인+감사 플래그 → 04 (D-04)
             │            └ OCR 장애 → manual 안내 (D-25)
             │        04: [이 자리(A3) 예약하기] 확정 1회 → 예약 완료 화면(D-09)
             │             └ 409 SEAT_TAKEN → "먼저 예약됐어요" + 다른 빈자리 QR 안내
             │                                      └ 자리 비우기 / 방명록 쓰기 → 05
             └─ 방명록 보기 → 05 방명록
```

## API 정리 (결정 반영 — db-schema.md 대비 변경분 포함)

| API | 메서드 | 변경점 |
|---|---|---|
| `/api/status` | GET | `{ available_seats, today_visitors, is_full }` |
| `/api/shops` | GET | 변경 없음 |
| `/api/verify` | POST | 응답 3상태: `{ status: approved\|rejected\|retry, token?, reason_code?, verification_id }` — review 없음(D-05) |
| `/api/verify/status` | GET | **신설.** device_id 기준 당일 인증 상태 조회 (재진입 복원용. 폴링 용도 아님) |
| `/api/seats` | GET | 변경 없음. **D-26**: 그리드 렌더용이 아니라 *찍은 좌석 하나*의 상태(available/taken/closed) 판정용으로 소비(클라이언트가 seat id로 필터) |
| `/api/reserve` | POST | 응답에서 `qr_token` 제거(D-09). `{ reservation_id, seat, expires_at }`. **D-26**: `seat_id`는 그리드 선택이 아니라 QR에서 온 값. API 자체는 무변경 |
| `/api/reservations/:id` | GET/DELETE | **신설.** 내 예약 조회(D-10) / 자리 비우기(D-09). device_id 소유 검증 |
| `/api/guestbook` | GET/POST | POST body에 `rating` 추가(D-17) |
| `/api/admin/*` | — | db-schema 목록 + `login`, 당일 통계(D-22) |

## localStorage 표준 규약 (전 파일 공통)

| 키 | 값 | 쓰는 곳 → 읽는 곳 |
|---|---|---|
| `stop-island:device-id` | UUID 문자열 (D-07) | 앱 부트 → 모든 API 헤더 |
| `stop-island:verify` | JSON `{ token?, verification_id, status, issued_date, shop_name? }` | 03 인증 → 02/04 가드·배너 |
| `stop-island:reservation-id` | reservation_id 문자열 (D-10) | 04 확정 → 01 배너·04 복구 |
| `stop-island:seat` | seat id 문자열 (D-26) | QR 부트(`?seat=`) → 01 메인 맥락·03→04 예약 확정. 최신 스캔이 덮어씀 |

원칙: localStorage는 캐시일 뿐, 단일 진실은 항상 서버(`GET /api/verify/status`, `GET /api/reservations/:id`). 불일치 시 서버 값으로 덮어쓰고 stale 키는 삭제한다.
