# 05 방명록 — 시나리오 (GB)

> **한 줄 요약:** 인증 없이 누구나 쓰는 익명 방명록 — 작성 폼(본문 500자·별점·맛집 태그 0~5개) 검증, 게시(rate limit 429 포함), cursor 기반 무한스크롤 목록, XSS·관리자 삭제·상점 비활성화 연동까지의 Given/When/Then 명세.

**커버 범위:**
- 작성 폼 검증 (빈 값 / 공백만 / 500·501자 경계 — D-18)
- 별점 선택·해제 (D-17, null 허용)
- 맛집 태그 바텀시트 (0~5개, 6개째 차단, 칩 표시/제거)
- 게시 (성공·폼 리셋·목록 갱신 / 실패 입력 보존 / 429 rate limit — D-19 / 중복 제출·두 탭 동시)
- 목록 (최초 로딩 / 빈 상태 / 실패+재시도 / cursor 무한스크롤 / 마지막 페이지 / 커서 안정성)
- 표시 (상대 시간, XSS 이스케이프, 비활성 상점 태그, 관리자 삭제 반영)

**저장소:** 방명록은 완전 익명(db-schema)이라 localStorage에 아무것도 저장하지 않는다. rate limit은 서버가 `X-Device-Id` 헤더(D-07)로만 판정한다. 이하 모든 시나리오에서 별도 명시 없으면 **localStorage 변화 없음**.

## 작성 폼 검증 (D-18)

관점 커버: 정상 GB-101 / 경계 GB-104, GB-105 / 오류 GB-102, GB-103, GB-106 / 동시성 — 해당 없음(검증은 로컬 단독 동작, 게시 경합은 GB-4xx가 커버) / 복구 GB-107

> 📌 결정: textarea에 `maxlength=500`을 걸고 **글자수 카운터 "N/500"** 을 우하단에 상시 표시한다. 500자 도달 시 카운터를 강조색으로 표시(입력은 이미 차단됨). 서버도 trim 후 1~500자를 독립 검증한다(클라이언트 우회 방어).
>
> 📌 결정: 작성 중 초안(draft)은 저장하지 않는다 — 새로고침하면 입력이 사라지는 것을 허용. 7일 팝업의 한두 문장 후기에 draft 보존은 과설계.

### 본문 입력 검증

#### GB-101 정상 입력으로 게시 가능 상태
- Given: 방명록 화면이 열려 있고 폼이 비어 있다 ([게시하기] disabled)
- When: "오늘 여기서 쉬다 갑니다"를 입력한다
- Then: [게시하기]가 활성화되고 카운터가 "13/500"으로 표시된다
- And: 서버 호출 없음

#### GB-102 빈 값
- Given: 폼이 비어 있다
- When: 아무것도 입력하지 않은 상태를 유지한다
- Then: [게시하기]는 disabled — 빈 값 제출 자체가 불가능
- And: 서버 호출 없음

#### GB-103 공백만 입력
- Given: 폼에 공백·줄바꿈만 입력했다 ("&nbsp;&nbsp;\n&nbsp;")
- When: 입력값을 trim 검증한다
- Then: trim 결과가 0자이므로 [게시하기]는 disabled 유지 (카운터는 원문 길이 기준 표시)
- And: 우회하여 공백만 POST해도 서버가 400 `CONTENT_EMPTY` 반환, row 미생성

#### GB-104 정확히 500자 (경계)
- Given: 정확히 500자(trim 후)를 입력했다
- When: [게시하기]를 탭한다
- Then: 게시 성공 — guestbook_entries에 content 500자 row 생성. 카운터는 입력 중 "500/500" 강조 표시였음
- And: 목록 갱신은 GB-401과 동일

#### GB-105 501자 시도 (경계)
- Given: 500자가 이미 입력돼 있다
- When: 1자를 더 입력하거나 501자 텍스트를 붙여넣는다
- Then: maxlength로 500자에서 잘려 입력된다(붙여넣기도 500자까지만 반영). 서버 요청 없음
- And: API를 직접 호출해 501자를 보내면 서버가 400 `CONTENT_TOO_LONG` 반환, row 미생성

#### GB-106 검증 우회 POST (오류)
- Given: 클라이언트를 거치지 않고 API를 직접 호출한다
- When: `POST /api/guestbook`에 빈 content / 501자 / shop_tags 6개를 보낸다
- Then: 각각 400 `CONTENT_EMPTY` / `CONTENT_TOO_LONG` / `TOO_MANY_TAGS` 반환. row·태그 모두 미생성
- And: 화면 영향 없음

#### GB-107 작성 중 새로고침 (복구)
- Given: 본문을 쓰고 태그 2개, 별점 4점을 골라 둔 상태다
- When: 브라우저를 새로고침한다
- Then: 폼이 빈 초기 상태로 렌더된다 — draft 미보존(📌 위 결정). 목록은 정상 재조회
- And: 서버 변화 없음

## 별점 (D-17)

관점 커버: 정상 GB-201 / 경계 GB-203(null 허용) / 오류 GB-204 / 동시성 — 해당 없음(로컬 UI 단독) / 복구 — 해당 없음(GB-107이 커버, draft 미보존)

> 📌 결정: 별점 해제 UX — **현재 선택된 점수의 별을 다시 탭하면 해제**되어 null로 돌아간다(예: 4점 상태에서 4번째 별 재탭 → 해제). 별점 UI는 [별점 추가] 버튼 탭 시 인라인으로 펼쳐지는 별 5개 행으로 하고, 선택 시 버튼 라벨이 "★ 4점"으로 바뀐다.

### 선택과 해제

#### GB-201 별점 선택
- Given: 별점 미선택 상태다
- When: [별점 추가] 탭 → 별 5개 중 4번째 별을 탭한다
- Then: 별 1~4가 채워지고 버튼 라벨이 "★ 4점"으로 바뀐다. 게시 시 body에 `rating: 4` 포함
- And: 서버 호출 없음(게시 전까지 로컬 상태)

#### GB-202 별점 변경
- Given: 4점이 선택돼 있다
- When: 2번째 별을 탭한다
- Then: 별 1~2만 채워지고 라벨 "★ 2점"
- And: 서버 호출 없음

#### GB-203 별점 해제 → null 게시 (경계)
- Given: 4점이 선택돼 있다
- When: 4번째 별을 다시 탭해 해제하고(📌 위 결정) 본문만으로 게시한다
- Then: body에 `rating: null` — guestbook_entries row의 `rating`이 NULL로 저장된다. 목록 카드에는 별점 영역 자체가 표시되지 않는다
- And: 별점은 선택 사항이며 게시 가능 조건에 영향 없음

#### GB-204 범위 밖 rating 우회 POST (오류)
- Given: API를 직접 호출한다
- When: `rating: 0` 또는 `rating: 7`을 보낸다
- Then: 서버가 400 `INVALID_RATING` 반환 (DB CHECK 1~5와 이중 방어), row 미생성
- And: 화면 영향 없음

## 맛집 태그 바텀시트

관점 커버: 정상 GB-301, GB-302, GB-304 / 경계 GB-303 / 오류 GB-305 / 동시성 — 해당 없음(로컬 선택 상태, 상점 목록은 읽기 전용) / 복구 GB-305(재시도)

### 상점 선택 (0~5개)

#### GB-301 바텀시트 열기
- Given: 폼 영역이 보인다
- When: [맛집 추가]를 탭한다
- Then: 바텀시트가 열리고 `GET /api/shops`(`is_active=true`, sort_order 정렬) 결과가 체크 가능한 리스트로 표시된다. 로딩 중에는 시트 내 스피너
- And: 서버 상태 변화 없음

#### GB-302 태그 선택과 칩 표시
- Given: 바텀시트가 열려 있다
- When: "막걸리계보", "카페 A" 2곳을 체크하고 시트를 닫는다
- Then: [맛집 추가] 버튼 아래에 선택 칩 2개("막걸리계보 ×", "카페 A ×")가 표시되고 버튼 라벨에 "맛집 추가 (2)" 카운트 표시. 게시 시 `shop_tags: ["makgeolli-gyebo", "cafe-a"]` 포함
- And: 서버 호출 없음(게시 전까지 로컬)

#### GB-303 6개째 선택 시도 (경계)
- Given: 이미 5곳이 체크돼 있다
- When: 6번째 상점을 체크하려 한다
- Then: 체크되지 않고 "맛집 태그는 5개까지 붙일 수 있어요" 안내가 시트 내에 표시된다 (D-18: 0~5개)
- And: 우회 POST 6개는 GB-106대로 서버 400 `TOO_MANY_TAGS`

#### GB-304 칩으로 태그 제거
- Given: 칩 2개가 표시돼 있다
- When: "카페 A" 칩의 ×를 탭한다
- Then: 칩이 사라지고 카운트가 (1)로 갱신, 바텀시트를 다시 열면 해당 상점 체크 해제 상태
- And: 서버 호출 없음

#### GB-305 상점 목록 로딩 실패와 재시도
- Given: 바텀시트를 열었는데 `GET /api/shops`가 실패했다
- When: 시트 내 "목록을 불러오지 못했어요" + [다시 시도]에서 재시도가 성공한다
- Then: 상점 리스트가 정상 표시된다. 실패 상태에서도 시트를 닫고 태그 없이 게시하는 데는 지장 없음
- And: 서버 상태 변화 없음

## 게시 (POST /api/guestbook)

관점 커버: 정상 GB-401 / 경계 GB-403(rate limit 경계) / 오류 GB-402 / 동시성 GB-404, GB-405 / 복구 GB-402(입력 보존 후 재시도)

> 📌 결정: 게시 성공 후 목록 갱신은 **optimistic insert 없이 첫 페이지 재조회**(`GET /api/guestbook?limit=10`, cursor 초기화)로 한다. 스크롤을 목록 최상단으로 이동시켜 내 글이 맨 위에 보이게 한다. 근거: POST 응답에 created_at이 없고, 재조회가 커서 상태 꼬임 없이 가장 단순.

### 제출

#### GB-401 게시 성공
- Given: 본문 "국밥 최고" + 태그 1개 + 별점 5점이 입력돼 있다
- When: [게시하기]를 탭하고 `POST /api/guestbook`이 `{ entry_id }`로 성공한다
- Then: 화면 — 폼 전체 리셋(본문·태그 칩·별점·카운터 "0/500"), 목록 첫 페이지 재조회 후 최상단으로 스크롤되어 내 글이 "익명 방문자 · 방금 전"으로 보인다. 서버 — guestbook_entries row 1건 + guestbook_shop_tags row 1건 생성
- And: localStorage 변화 없음(익명)

#### GB-402 게시 실패 — 입력 보존 (오류·복구)
- Given: GB-401과 같은 입력 상태다
- When: POST가 네트워크 오류/5xx로 실패한다
- Then: "게시에 실패했어요. 다시 시도해주세요" 에러 표시. **본문·태그·별점 모두 보존**되고 [게시하기] 재활성화 — 그대로 재시도 가능
- And: 서버 row 미생성 (5xx 시 서버는 트랜잭션 롤백으로 entries/tags 부분 생성 금지)

#### GB-403 rate limit 429 (D-19)
- Given: 같은 기기(device_id)로 40초 전에 글을 게시했다
- When: 새 글을 [게시하기]로 제출한다
- Then: 서버가 429 `RATE_LIMITED` 반환 → "잠시 후 다시 써주세요" 안내 표시. 입력값은 GB-402와 동일하게 보존. 1분 경과 후 재시도하면 성공
- And: 서버 row 미생성

#### GB-404 더블탭 중복 제출 방지 (동시성)
- Given: 유효한 입력이 있다
- When: [게시하기]를 빠르게 두 번 탭한다
- Then: 첫 탭 즉시 버튼 disabled+스피너 — 요청은 1회만 발생, row 1건만 생성
- And: 방어가 뚫려 2회 요청이 나가면 두 번째는 D-19 rate limit에 걸려 429 — 중복 row 없음

#### GB-405 두 탭 동시 게시 (동시성)
- Given: 같은 기기에서 방명록을 두 탭에 열고 각각 글을 써 뒀다
- When: 두 탭이 거의 동시에 게시한다
- Then: 먼저 도착한 요청만 성공하고, 나중 요청은 429 `RATE_LIMITED`(1분 1회 — D-19) → 해당 탭은 GB-403과 동일하게 입력 보존 + 안내
- And: 서버 row는 1건만 생성

## 목록 조회 & 무한스크롤

관점 커버: 정상 GB-501, GB-504 / 경계 GB-502, GB-505 / 오류 GB-503 / 동시성 GB-506 / 복구 GB-503(재시도), GB-508

> 📌 결정: cursor는 **`created_at` + `id` 복합 keyset 커서**(불투명 문자열로 인코딩)로 한다. offset이 아니므로 스크롤 도중 새 글이 앞에 추가되거나 관리자가 글을 삭제해도 다음 페이지에 중복·누락이 생기지 않는다.

### 페이지네이션

#### GB-501 최초 로딩 성공
- Given: 서버에 글이 23건 있다
- When: 방명록 화면에 진입해 `GET /api/guestbook?limit=10`이 성공한다
- Then: 스켈레톤이 걷히고 최신순 10건이 카드로 렌더된다(익명 방문자 · 상대 시간 · 본문 · 별점 · 태그 칩). 응답의 `cursor`는 non-null
- And: 서버 상태 변화 없음

#### GB-502 빈 상태 (경계)
- Given: 글이 0건이다
- When: 최초 로딩이 성공한다
- Then: "아직 등록된 방명록이 없어요. 첫 후기를 남겨주세요!" 빈 상태 문구 표시, `cursor: null`
- And: 서버 상태 변화 없음

#### GB-503 최초 로딩 실패와 재시도 (오류·복구)
- Given: 네트워크 단절 상태로 진입했다
- When: GET이 실패한다 → "방명록을 불러오지 못했어요" + [다시 시도] 탭 → 성공한다
- Then: 실패 시 목록 자리에 에러+재시도 버튼(작성 폼은 정상 사용 가능), 재시도 성공 시 GB-501과 동일 렌더
- And: 서버 상태 변화 없음

#### GB-504 다음 페이지 로드
- Given: 10건이 렌더돼 있고 cursor가 있다
- When: 리스트 하단에 도달한다(Intersection Observer)
- Then: 하단 스피너 → `GET /api/guestbook?cursor={cursor}&limit=10` → 11~20번째 글이 이어 붙는다. 중복 카드 없음
- And: 서버 상태 변화 없음

#### GB-505 마지막 페이지 (경계)
- Given: 23건 중 20건이 렌더돼 있다
- When: 하단 도달로 3번째 페이지를 로드한다
- Then: 3건이 붙고 응답 `cursor: null` → "마지막 방명록이에요" 종료 표시, 이후 하단 도달해도 추가 요청 없음
- And: 서버 상태 변화 없음

#### GB-506 스크롤 중 새 글 작성됨 — 커서 안정성 (동시성)
- Given: 내가 1페이지(10건)를 본 상태에서 다른 사용자가 새 글 2건을 게시했다
- When: 내가 하단 도달로 다음 페이지를 요청한다
- Then: keyset 커서(📌 위 결정)가 "1페이지 마지막 글보다 오래된 글"만 반환하므로 **중복·누락 없이** 기존 11~20번째 글이 로드된다. 새 글 2건은 새로고침 또는 내 게시(GB-401) 시 첫 페이지 재조회로 보인다
- And: 서버 상태 변화 없음

#### GB-507 상대 시간 표시 (경계)
- Given: 글들의 created_at이 각각 30초 전 / 10분 전 / 3시간 전 / 2일 전이다
- When: 목록이 렌더된다
- Then: "방금 전" / "10분 전" / "3시간 전" / "2일 전"으로 표시된다(1분 미만은 "방금 전"). 기준은 서버 created_at(ISO 8601)의 클라이언트 상대 변환
- And: 서버 상태 변화 없음

#### GB-508 게시 직후 새로고침 (복구)
- Given: GB-401 직후다
- When: 브라우저를 새로고침한다
- Then: 폼은 초기 상태, 목록 첫 페이지에 내 글이 그대로 최상단에 보인다 — 익명이라 "내 글" 표시는 없다
- And: 서버·localStorage 변화 없음

## 표시·보안·관리 연동

관점 커버: 정상 GB-603 / 경계 GB-602 / 오류(보안) GB-601 / 동시성 GB-604 / 복구 — 해당 없음(단순 읽기 표시 영역, 실패 시 GB-503 재시도 경로와 동일)

> 📌 결정: 태그된 상점이 이후 관리자에 의해 비활성화(`is_active=false`)되어도 **기존 글의 태그 칩은 상점명 그대로 계속 표시**한다. 태그는 작성 시점의 사실 기록이며, 목록 조회는 is_active 필터 없이 shops를 join한다. 비활성화 영향은 신규 작성 바텀시트(GET /api/shops)에서만 나타난다(목록에서 제외).

### 렌더링과 어뷰징 대응

#### GB-601 XSS 시도 입력
- Given: 본문에 `<script>alert(1)</script><img src=x onerror=alert(2)>`를 입력해 게시했다
- When: 목록에서 해당 카드가 렌더된다
- Then: 스크립트가 **실행되지 않고** 입력한 문자열이 그대로 텍스트로 보인다(Svelte 텍스트 바인딩 기본 이스케이프, `{@html}` 사용 금지). 서버는 원문을 그대로 저장(이스케이프는 렌더 계층 책임)
- And: 알림창·네트워크 부작용 없음

#### GB-602 비활성화된 상점이 태그된 글 (경계)
- Given: "카페 A"가 태그된 글이 있고, 이후 관리자가 카페 A를 `is_active=false`로 바꿨다
- When: 목록을 조회한다
- Then: 해당 글의 "카페 A" 칩은 그대로 표시된다(📌 위 결정). 신규 작성 바텀시트에는 카페 A가 나타나지 않는다
- And: guestbook_shop_tags row는 유지

#### GB-603 관리자가 삭제한 글 반영
- Given: 목록에 보이던 글을 관리자가 `DELETE /api/admin/guestbook/:id`로 삭제했다
- When: 사용자가 목록을 새로고침하거나 다음 페이지를 로드한다
- Then: 해당 글이 응답에서 빠져 화면에서 사라진다. keyset 커서 특성상 삭제로 인한 다른 글의 중복·누락 없음. 서버 — entries row 삭제 + shop_tags CASCADE 삭제
- And: 이미 렌더된 화면에서 실시간으로 지우지는 않는다(방명록은 폴링 없음 — 다음 조회 시 반영이면 충분)

#### GB-604 삭제와 페이지 로드의 경합 (동시성)
- Given: 내가 다음 페이지를 요청하는 순간 관리자가 그 페이지 범위의 글 1건을 삭제했다
- When: `GET /api/guestbook?cursor=...`가 처리된다
- Then: 삭제 시점에 따라 해당 글이 포함되거나 빠질 뿐, 응답은 정상 200이며 커서 연속성이 깨지지 않는다(keyset은 삭제에 안전). 화면 오류 없음
- And: 서버 상태 변화는 삭제 자체뿐
